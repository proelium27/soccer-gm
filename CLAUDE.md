# soccer-gm

A client-side, BBGM-style soccer management sim (TypeScript). `SOCCER_GM_SPEC.md` is the full design brief and M0-M6 build order.

## Git workflow
- Always keep work committed: after making code changes, commit them locally with a clear message rather than leaving the working tree dirty.
- Always keep the remote in sync: after committing, push to GitHub (`origin`) so local `main` and GitHub `main` never drift apart.
- Merging isn't finished until it's in both places: when a PR is merged on GitHub, also pull that merge into the local `main` branch (`git checkout main && git pull`) so the local checkout matches GitHub. Don't consider a change "done" until local `main` and GitHub `main` both have it.
- Only skip this (leave changes uncommitted/unpushed/unpulled) if the user explicitly asks you to hold off.

- **M2 — Season loop, persistence, minimal UI**: done, merged. Vite scaffold, IndexedDB persistence, web worker, team identity/scheduling/league state, `simThrough`, initial UI pages/routing.
- **M3 — Box scores**: done, merged (`6a22da7`). Per-player attribution (shooter/assister/tackler/GK), `simMatchDetailed`, fleshed-out `SeasonStats`, box score + stat leaders pages.
- **M4 — Dynasty mechanics**: done, merged (`e49c7b7`, progression reworked in `6a79eb9`). Progression (BBGM-style age curve — see `src/core/players/progression.ts`), aging, retirement, youth intake, free agency, contracts, offseason orchestrator. Manually browser-verified.
  - Known gap fixed later: potential could pile up at the 99 clamp; replaced with a soft asymptotic ceiling (`bd24519`).
  - Progression retuned alongside the M1 OVR rebalance above (2026-07-10), then corrected further (same date) after a full-dynasty audit found the first pass only delayed, rather than fixed, unbounded league-wide OVR inflation:
    - **Potential is no longer a growth input.** Per the real BBGM manual, potential shouldn't influence development at all — it's a scout's *estimate*, computed by simulating a player's future career arc forward and reading off the 75th percentile of the simulated peak ovr (`estimatePotential`). The old design re-rolled potential from the player's *current* ovr every offseason and used the `(potential - ovr)` gap to scale growth, which created a ratchet: a lucky season raised the ceiling, sustaining more growth next season, compounding indefinitely. Actual growth now depends only on age/rating-group and noise, matching the manual exactly.
    - **`SKILL_AGE_SHIFT` (-3→-1.5) and `GK_AGE_SHIFT` (-3→-0.5) were nearly zeroed out.** Even after removing the potential ratchet, empirical dynasty simulation (25+ seasons via the real `simOffseason`) still showed severe inflation. Root cause: the old shifts made the *survival-weighted expected lifetime rating delta* for skill-group ratings net **positive** (+10.35 over a realistic career, worst for GK/CM/AM/DM whose ovr is skill-dominated) even accounting for retirement — every surviving player was expected to keep improving essentially forever. Retuned so every position's expected lifetime arc is ~0 or slightly negative, like real football aging.
    - **Youth intake no longer anchors to the club's current roster average.** This was the dominant driver: `generateYouthIntake`'s quality base was `teamAvgOvr - YOUTH_BASE_OFFSET`, so once a roster drifted upward (from any source of noise), next season's youth were generated relatively stronger too, permanently ratcheting the anchor upward with no reversion. Fixed by anchoring youth intake to a new fixed-at-generation `academyBase` field on each team (`LeagueTeam`/`StoredTeam`, migrated for old saves in `src/db/migrate.ts`) — the same `LEAGUE_BASE + strength target` used to generate that team's initial roster, never touched again.
    - Verified via a from-scratch 40-season dynasty simulation (real `simOffseason`, not a proxy): AI-rostered OVR distribution now reaches a genuine stationary equilibrium by ~season 17 (mean ~58, 80+ under 2%, 90+ at 0%) instead of climbing without bound (the pre-fix state reached 96.6% at 80+ and 30.9% at 90+ by season 25 alone).
- **M5 — Match texture**: done, merged (`20ee78d`, PR #10). Cards (yellow/second-yellow/straight red) with a once-per-side man-down penalty, per-player fatigue + AI substitutions at 60'/75' (attacking sub when trailing), corners, penalties (taker-vs-keeper conversion, saved/off-target miss split), in-match injuries with multi-matchday recovery persisted through `simThrough`, event-weighted stoppage time, and composite re-rolls from the on-pitch XI after subs/red cards/injuries (spec §4). Deferred follow-ups: injuries have no roster-page UI (players silently sit out), and `simMatch`/`simMatchDetailed` share ~100 duplicated foul/set-piece lines worth extracting.
- **M6 — Depth, finance slice** (chosen from the §7 "pick by fun" menu): **foundation done, merged** (`df4ed88`, PR #15). Design (budgets/scouting/hype/contracts/transfer valuation) was finalized with the user in a prior conversation (details beyond this summary live in a Google Doc the user maintains — ask if you need the full spec). Landed: equal base club budgets with season-end settlement (base allocation + tiered rank payouts + damped hype→revenue, minus wages/scouting spend) tuned so AI clubs never run deficits, hype tracking driven by season performance, scouting-spend slider, an OVR/age/contract-driven transfer valuation formula with scouting-scaled noise, and an old-save migration backfilling the new `StoredTeam` fields. **Phases 3-7 done, merged (PR #17)**: two transfer windows per season (summer = offseason + matchdays 1-4, winter = matchdays 18-22 with deadline day as the "Sim to Transfer Deadline" landing spot; window identity survives the offseason season rollover — see `transferWindowState().season`), a Recommended Transfers page (5-10 for-sale players near the user's level within budget, scouting-scaled noise), user-initiated offer/counter-offer negotiation against hidden window-stable reservation prices (lowballs/non-improving repeats/patience collapse talks for the window), and one-button contract extensions/free-agent signings (ovr-based salary shown as a weekly wage, age-based length).

A hard `ROSTER_CAP` (30, `src/core/constants.ts`) was added after M6 to block free-agent signings and transfer buys once the user's squad is full (Roster/Incoming Talent/Transfers pages show a "x/30" indicator); youth intake and AI free agency are unaffected (AI is still trimmed back to `ROSTER_COMPOSITION`'s 25 every offseason, and youth intake can still push a team briefly over the cap until a slot opens).

The Starting XI is no longer purely auto-selected: the Roster page now supports drag-and-drop swaps between the Starting XI and bench, persisted per-team as `StoredTeam.starters` (11 pids, migrated to `null` for old saves) and consumed by match sim via a new `resolveXI()` helper (`src/core/lineup/resolveXI.ts`) that transparently falls back to the existing `selectXI()` auto-pick whenever `starters` is unset or invalid per `isValidStarters` (stale pid after a release/sale/injury, duplicate pids, or a GK/outfield slot mismatch). Only the user's team ever has `starters` set — AI teams always auto-select. The formation itself is still hardcoded to 4-3-3; there's no formation picker yet.

A post-review hardening batch (bug-hunt sweep, 2026-07-11) fixed several correctness issues without changing any game design: all league-mutating UI actions are now serialized through a single promise chain in `LeagueContext` (previously two quick actions could compute from the same stale snapshot and silently lose the first write — e.g. sign two free agents fast and one vanished while both budgets were charged), the sim overlay now persists its result *before* closing, manual releases enforce the same per-position depth floor the transfer market's `isForSale` uses (shared `keepsDepthFloor` in `src/core/freeAgency.ts` — previously a user could release their entire roster, which crashed the match engine and bricked the save), GK↔outfield lineup swaps are rejected at both the UI and the action layer, the sim worker rejects its promise on a crash instead of hanging the "Simming" overlay forever, lingering injuries are healed at the offseason rollover, and `migrateLeague` tolerates pre-M3 saves whose played matches have no `boxScore`.

Multi-league support (BBGM-style) was added after M6: the IndexedDB `leagues` store already supported multiple saves keyed by `lid` (`src/db/leagueDb.ts`), but the UI/state layer previously auto-loaded whichever save was first in the store. Now a `/leagues` landing page (`src/ui/pages/Leagues.tsx`) lists all saves with Enter/Delete, and `LeagueContext` (`src/ui/context/LeagueContext.tsx`) tracks the active save via a `localStorage` pointer (`src/db/activeLeague.ts`) instead of always loading `list[0]`. A "Switch League" link in the sidebar clears the pointer and returns to `/leagues`; existing league-scoped routes/pages/links were untouched. New leagues are now named after the chosen club (previously every save was hardcoded to "My League", which was fine for one save but indistinguishable once multiple exist).

**Club identities are fictional and user-customizable** (2026-07-12): the default `CLUBS` list (`src/core/teams/clubs.ts`) was switched from real Premier League clubs to 20 invented English-flavored clubs (trademark caution — real club names are PL-licensed marks; deliberately fictional rather than PES-style soundalikes, per user decision), and a "Customize Teams" button on each save in `/leagues` opens an editor to rename any club and change its abbreviation/colors per save (`customizeTeamsAction` in `LeagueContext` + pure `applyTeamIdentities` in `src/core/teams/customize.ts`; renaming the user's own club also renames the league while the two names still match). Two consequences to know about: `migrateLeague` no longer re-stamps identities from `CLUBS` on every load (that would revert customizations) — it only backfills missing fields — and existing saves therefore keep whatever names they were created with, including real PL names from the earlier era; use Customize Teams to change them.

**Known gaps carried forward:** AI↔AI transfers and inbound offers for the user's players now exist (see "AI General Manager realism" below); user's own roster is never auto-trimmed by AI logic (by design) — the `ROSTER_CAP` above blocks further *acquisitions* once full but doesn't force a release; injured players sit out without any roster UI indication; no scouting uncertainty beyond the M6 valuation noise (potential itself is still fully accurate/visible to the user).

## Open design decisions (from the 2026-07-11 bug hunt)

A full-codebase review (PR #34 fixed the mechanical bugs) left four findings that need a **design call with the user before coding** — each has more than one reasonable fix. Whoever picks these up: discuss options first, don't just implement.

1. **Winter-sale wage dodge.** Season wages are charged once, at offseason settlement, against whoever holds the player *at that moment* (`wageBill` over the current roster in `simOffseason` step 3.5). Selling a high earner in the winter window therefore dodges his entire season wage bill; the buyer — who had him for half a season — pays all of it. Exploitable by the user. Options: pro-rate wages by ownership stints, charge the seller their share at transfer execution, or accept it as a deliberate quirk and document it.
2. **The "no club ever runs a deficit" tuning breaks at the 30-man cap.** It was only ever proven for the 25-man `ROSTER_COMPOSITION`: worst case 30 players × `SALARY_PER_OVR` (20k) × 99 OVR = $59.4M wages + `SCOUTING_SPEND_MAX` ($20M) = $79.4M > `BASE_SEASON_BUDGET` ($75M), so a stacked full user squad with a low finish goes negative. Options: retune the budget constants for 30, cap scouting spend relative to the wage bill, or decide user deficits are actually allowed (the invariant was originally about AI clubs) and add debt UI/consequences.
3. **Youth intake positions don't match roster needs.** `generateYouthIntake` draws positions uniformly (12.5% each) while rosters need CB/FB/CM at 16% each, so over long dynasties the league drifts structurally short of those positions and `runAIFreeAgency` silently gives up when a position's free-agent pool is empty. Likely fix: weight the youth position draw by `ROSTER_COMPOSITION` — but confirm the change to rng-stream consumption is acceptable (it shifts every downstream roll in that offseason).
4. **Mid-season transfer stat attribution.** `SeasonStats` is one running total per player per season with no per-club stint, and Leaders/Roster attribute the whole line to the player's *current* club — buy a 10-goal striker at the winter deadline and your team gets display credit for all 10. Options: per-stint stats (schema change + save migration) or keep the single total and add a "joined mid-season" marker in the UI.

Lower-severity leftovers from the same review, no design input needed, just not yet done: no guard anywhere against fielding fewer than 11 when a roster drops below 11 healthy players (`selectXI` silently skips slots — needs a rule: emergency call-up, forfeit, or hard block); `pickReplacement` in `matchSim.ts` would hand an outfielder back as a "GK replacement" if GKs could ever be subbed/injured/carded (currently unreachable — a landmine for whoever adds GK injuries).

A FotMob-style **Match Rating** stat was added post-M6. `computeMatchRating()` (`src/engine/matchRating.ts`) scores each player's `PlayerMatchLine` 0.0–10.0 from a fixed 6.0 baseline using a position-weighted matrix (GK/DEF/MID/FWD columns for goals, assists, shots on target, tackles, saves, clean sheets, goals conceded, cards) supplied by the user, applied only to the stats this engine's box score actually tracks — pass accuracy, big-chances-created, own goals, errors-leading-to-goal, and a separate penalty-save flag aren't modeled since the sim has no underlying data for them. A player's swing away from 6.0 is damped by `minutesPlayed / 90` so cameo subs can't post extreme ratings off a couple of touches. `minutesPlayed` itself is new too: reconstructed from each match's substitution/red-card/injury event clocks in `simMatchDetailed` (`src/engine/matchSim.ts`), not tracked directly during the sim loop. Rating and minutes are stored per-line on `PlayerMatchLine` (shown on the Box Score page) and rolled up into a new `SeasonStats.avgRating` (via `ratingSum`), surfaced as a sortable column on the Stat Leaders page. `src/db/migrate.ts` backfills both fields for pre-existing saves — historical box scores get a neutral 0 minutes / 6.0 rating since the clock data needed to reconstruct them doesn't survive in old saves.

A **Finance** page (`src/ui/pages/Finance.tsx`, sidebar under Team, linked from the Dashboard's Finances card) was added post-M6 to surface the existing M6 finance model in one place: current budget/hype plus the scouting-spend slider (same control as the Dashboard's), a season-settlement breakdown mirroring `runOffseason`'s math exactly (base + prize tier by rank + damped hype revenue − wage bill − scouting spend; labeled a projection during the regular season, the final settlement during the offseason), the full per-player wage-bill table, the user's transfer history with all-time spend totals, and a league-wide budget/hype/wage-bill table. Beyond an `ordinal()` helper moved into `src/ui/format.ts`, the page itself is pure display; the Dashboard's Finances card also gained a "Wages due at season end" line.

**Wages were reworked in the same PR** (2026-07-11): the flat 20k-per-ovr placeholder salary became a Premier-League-calibrated cubic — weekly wage = `WAGE_WEEKLY_MIN + WAGE_WEEKLY_COEFF × (ovr − WAGE_OVR_FLOOR)³` times a deterministic per-signing ±15% roll (`seasonSalaryForOvr` in `src/core/contracts.ts`, used by contract terms, AI free agency, and player generation). ~£22k/wk at 60 ovr → ~£314k/wk at 90, versus the old 23k→34k range. `BASE_SEASON_BUDGET` rose 75M→95M to keep AI clubs deficit-free: the old "base covers 25×99-ovr wages" invariant is unreachable under the cubic, so the tested guarantee is now "base covers `WAGE_SAFE_SQUAD`", a benchmark shaped from 25-season dynasty audits (max observed AI wage bill ~86M, settlement margins never below +32M). A user hoarding an elite ROSTER_CAP squad *can* now out-wage the base — the Finance page projects the shortfall (documented gap, sharper than before). Old saves keep their existing cheap contracts until players re-sign at the new rates. **Wage timing also changed**: wages are now paid up front at each season's start (`chargeSeasonStart` — base allocation in, squad wages out, applied at league creation and in `simOffseason` on the finalized new-season rosters), while season-end settlement (`settleSeasonEnd`) handles only prize + hype − scouting. A new club therefore starts with `BASE_SEASON_BUDGET − its initial wage bill` (~50-65M) instead of a full 95M, in-season cash is genuinely spendable, and mid-season acquisitions (transfer buys / free-agent signings during the regular phase) charge the player's full season salary at acquisition on top of any fee.

One tuning note for later: this engine's `tackles` field is a catch-all defensive-action count that can run into the high teens per match (unlike a real match's ~5-8 tackles), so at the given DEF weight (0.2/tackle) a scoreless center-back can rating-farm into the 9s. Worth revisiting the tackle weight if that shows up as a recurring pattern once more seasons are played.

## AI General Manager realism (evaluation-driven, in progress)

A multi-phase effort to replace the AI's rule-scripted behavior with a Basketball-GM-style evaluation system, where a club's strategic direction and how it values any given player both *emerge* from its current state rather than from hand-authored per-club scripts (full brief: the user's "AI General Manager Philosophy" doc — ask for it). Planned spine: (1) evaluation core, (2) AI↔AI transfer market, (3) AI inbound offers for the user's players, (4) proactive contract renewals, (5) imperfect/scouting-noised decisions. (Phase 4 was originally scoped as "contract renewals, sell-at-peak, replace-aging"; sell-at-peak turned out to already exist as an emergent effect of phase 2's transfer market, and replace-aging turned out not to need a separate mechanic — see the phase-4 bullet below.)

- **Phase 1 — evaluation core: done (this PR), no behavior change yet.** New `src/core/ai/` module, not yet consumed anywhere in the sim (so no observable change; expect the constants to be retuned once phases 2+ actually run on top of it). Two pieces:
  - `clubContext.ts` — `deriveLeagueContexts(league)` computes, for every club, a `ClubContext` derived entirely from current state: squad strength (mean ovr of best `AI_SQUAD_STRENGTH_COUNT`), mean age, per-position depth/best-ovr, and two league-normalized scalars — **ambition** (win-now pressure: weighted blend of wealth/fame/squad-strength/form) and **frugality** (financial caution: inverse of relative wealth) — plus a display-only `StrategicDirection` label. Per the user's call, philosophy is *derived from state*, not stored, so there's no schema change and no save migration.
  - `evaluate.ts` — `valueToClub(player, ctx)` / `evaluatePlayerForClub(...)` layer three club-relative multipliers on top of the existing market value (`trueTransferValue`): **positional need** (depth vs `ROSTER_COMPOSITION` × upgrade over the incumbent), **timeline** (age × ambition — win-now clubs pay up for prime, developer clubs for youth), and **affordability** (deal cost vs budget, penalty scaled by frugality). Returns a full breakdown for a future "why did they value him at X" UI. This is the primitive phases 2-5 build on (buy if value ≥ asking, sell if value < market, etc.). All tuning constants live in the `AI evaluation core` section of `src/core/constants.ts`.

- **Phase 2 — AI↔AI transfer market: done.** `src/core/ai/transferMarket.ts`'s `runAITransferMarket(...)` has AI clubs trade with each other, driven entirely by phase-1 `valueToClub` (the user's club is excluded from both sides — inbound offers for the user's players are phase 3). The whole market is one comparison applied everywhere: **a player's reservation price is his value to his *current* club, and he moves to whichever other club values him enough more to clear that reservation and afford the fee** (fee lands `AI_MARKET_FEE_SHARE` of the way between the two valuations; money conserved). Surplus dumping, sell-at-peak (an aging star's keep-value drops below his market price → available), and needs-based buying all emerge from that single rule — no scripted "if rebuilding…" branches. Guardrails: a club only shops players it values at ≤ `AI_MARKET_AVAILABILITY` × market (won't auction an irreplaceable core player), buyers need a `AI_MARKET_MIN_SURPLUS` margin to bother, per-club `AI_MARKET_MAX_BUYS`/`MAX_SELLS` caps per window, live enforcement of the depth floor (`keepsDepthFloor`), `ROSTER_CAP`, and affordability. Clubs hold back a **cash reserve** rather than blowing their whole budget: a buyer spends only the surplus above `reserveFraction × live budget`, where the fraction runs from `AI_MARKET_RESERVE_FRACTION_MIN` (wealthiest/least-frugal clubs) up to `AI_MARKET_RESERVE_FRACTION_MAX` (poorest), so no club is ever spent down to £0 (dynasty-audited min budget ~£0.3M+ over 120 season-years, vs a literal £0 before the reserve). Deterministic: valuations snapshotted at window open, deals executed in descending-surplus order, only randomness a seeded per-buyer valuation jitter (`AI_MARKET_VALUE_JITTER`, a first taste of phase-5 scouting imperfection). Tuning constants in the `AI↔AI transfer market` section of `constants.ts`.
  - **Runs once per window, both windows.** Summer: a new step in `simOffseason` (after `trimRosterSurplus`, before season-start wages), valued for the upcoming season with the just-finished season's form. Winter: fired inside `simThrough`'s matchday loop the first time a batch reaches `WINTER_WINDOW_OPEN_MATCHDAY` (18), which is why `simThrough` now threads team/budget/transfer state through the loop instead of snapshotting `league.teams` once. A new `LeagueStore.winterMarketRunSeason` (migrated to `null`) guards the winter market to exactly once per season regardless of how the user batches matchdays.
  - **Verified** via a 40-season × 3-seed dynasty audit (real `simThrough`/`simOffseason`): no AI club ever goes into deficit, OVR stays at its prior equilibrium (mean ~58, 80+ <1%, 90+ 0% — the market causes no inflation), AI rosters stay ≤ `ROSTER_CAP`, and clubs make ~103 transfers/season league-wide. Two existing tests were re-pointed at the corrected invariants (AI clubs now spend, so budgets no longer only grow — the real invariants are AI solvency + a rising league-wide total; AI squads can now carry up to `ROSTER_CAP`, not just the 25-man composition). A **News Feed** page (`src/ui/pages/NewsFeed.tsx`, `/news`, sidebar under League) surfaces the whole league's transfer activity — every completed deal in `league.transfers` (AI↔AI included), newest first, grouped into per-season cards with window/player/from→to/fee columns, club/season filter dropdowns, and user-club deals row-highlighted. Pure display over existing state (no schema change).

- **Phase 3 — AI inbound offers for the user's players: done.** `src/core/transfers/inboundOffers.ts` mirrors phase 2's evaluation-driven design but with the user as the seller and full counter-negotiation (a design call made with the user: offer timing recomputed per-window like Recommended Transfers rather than fired once at window open; negotiation goes both ways rather than take-it-or-leave-it; own `/incoming-offers` page rather than a section on Transfers).
  - `inboundOfferCandidates(league)` scans the user's roster and, for each player, finds the single AI club that values him most via the same `valueToClub`/`ClubContext` primitive phase 2 uses, offering only if that club clears the player's worth to the user's *own* club (`AI_MARKET_MIN_SURPLUS`, reused) — capped to `INBOUND_OFFERS_MAX` most compelling offers, with the same fee-split (`AI_MARKET_FEE_SHARE`) and a per-player seeded jitter (`INBOUND_OFFER_VALUE_JITTER`) for opening-offer variety. A buyer's ceiling is capped by what it can actually spend without dipping into its `AI_MARKET_RESERVE_FRACTION_*` cash reserve (`buyerSpendable`) — otherwise a club with a high-but-unspendable valuation would dangle offers it could never actually close.
  - **Negotiation is symmetric to outgoing offers but mirrored**: `respondToAsk(ceiling, ask, priorAsks)` is the buyer's-side twin of `negotiation.ts`'s `respondToOffer` (every "higher is better" comparison flipped to "lower is better"), reusing the exact same `NEGOTIATION_LOWBALL_FACTOR`/`COUNTER_PADDING_*`/`NEGOTIATION_MAX_ROUNDS` tuning so both directions of the market haggle at the same pace. `acceptInboundOffer`/`rejectInboundOffer`/`counterInboundOffer` persist state in a new `LeagueStore.inboundOffers` array (migrated to `[]` for old saves), separate from the outgoing `negotiations` array since the user is the seller here, not the buyer. `executeTransfer` (previously private to `negotiation.ts`) is now exported and shared by both directions.
  - **UI**: a new `/incoming-offers` page (sidebar under Team, next to Transfers) lists live offers with Accept/Counter/Reject controls, an "Other Offers" section for negotiations that outlived the current candidate list (mirrors Transfers' "Other Negotiations"), and a "Sold This Window" section — an accepted offer sells the player immediately, so without this he'd just silently vanish from the list with no on-page confirmation.
  - Browser-verified end to end (reject, counter → AI raises its offer → accept, and immediate accept) via a real dynasty run to the winter window.

- **Phase 4 — proactive contract renewals: done.** `src/core/ai/renewals.ts`'s `runAIContractRenewals(...)` closes a gap phases 1-3 didn't touch: previously a contract just expired (`releaseExpiredContracts`) and the player hit the open free-agent pool with zero priority for his own club to keep him — `runAIFreeAgency` only reacts to positional shortfalls, not "this is my good player." Now, as a new step 0 in `simOffseason` (before contracts expire and before AI free agency/the summer transfer market run), any AI player entering his contract's final season is renewed on the spot if `valueToClub(player, ctx) ≥ contractTerms(player, nextSeason).salary × AI_RENEWAL_MARGIN` — reusing the phase-1 evaluation core and the same one-button `contractTerms`/`extendContract` helpers the user's own renew button already calls, so no new pricing or contract-length logic was needed. `valueToClub` already folds in affordability, so a cash-strapped club naturally lets pricier players walk without a second budget gate. The original phase-4 scope also named "sell-at-peak" and "replace-aging," but investigation found both already covered: sell-at-peak is an existing emergent effect of phase 2's transfer market (a keep-value that drops below market price makes a player available), and replace-aging needs no separate mechanic — an old, declining player simply fails the renewal bar, walks (or is sold earlier via the existing market), and the vacancy is filled by the same free-agency/transfer-market machinery that fills any other hole. Verified via a 40-season × 3-seed dynasty audit (real `simThrough`/`simOffseason`): zero AI deficits, ~7,400 renewals/seed over 40 seasons (meaningfully active, not inert), and the league-wide OVR equilibrium unchanged from the pre-phase-4 baseline (80+ under 0.2%, 90+ at 0%).
