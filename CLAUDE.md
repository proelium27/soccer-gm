# soccer-gm

A client-side, BBGM-style soccer management sim (TypeScript). `SOCCER_GM_SPEC.md` is the full design brief and M0-M6 build order.

## Git workflow
- Always keep work committed: after making code changes, commit them locally with a clear message rather than leaving the working tree dirty.
- Always keep the remote in sync: after committing, push to GitHub (`origin`) so local `main` and GitHub `main` never drift apart.
- Merging isn't finished until it's in both places: when a PR is merged on GitHub, also pull that merge into the local `main` branch (`git checkout main && git pull`) so the local checkout matches GitHub. Don't consider a change "done" until local `main` and GitHub `main` both have it.
- Only skip this (leave changes uncommitted/unpushed/unpulled) if the user explicitly asks you to hold off.

The game has a player-facing Manual page (`src/ui/pages/Manual.tsx`, sidebar under Help, `/manual`) written in the Basketball GM manual's style: single scrollable page, second-person, plain-spoken, with concrete numbers quoted from `src/core/constants.ts`. Keep new sections to that standard, and keep the "FAQ & Known Quirks" section honest about known gaps — it documents them for players instead of hiding them.

Retuned so every position's expected lifetime arc is ~0 or slightly negative, like real football aging.

Landed in two passes the same day:
    - First pass added `PROGRESSION_FORM_SD_YOUNG`/`_OLD` — one shared gaussian "form" roll per rating group (physical/skill) per player per season, added identically to every rating in that group before the existing per-rating noise. This alone survives the ovr averaging and produces real swings, but every season was an independent re-roll, so a player's trajectory was still a fresh coin flip each year rather than a consistent "bust" or "clean developer" arc — a follow-up user request.
    - Second pass added a genuinely persistent per-player trait: `developmentBias` (`progression.ts`) derives a fixed gaussian z-score deterministically from the player's `pid` via a salted hash (*not* drawn from the shared `rng` stream, so it can't shift other players' generated ratings/names — see the RNG-stream-order lesson), scaled by `PROGRESSION_BIAS_SD_YOUNG` and applied every season on top of the form roll. **This must taper to exactly 0 by peak age (`biasSdAt`, keyed off `BASE_AGE_CURVE_PEAK`), not just narrow toward retirement age like the other noise terms** — the first attempt tapered it like the others and gave a lucky player's bias a nonzero contribution all the way through decline years, reopening the exact unbounded-inflation failure mode `SKILL_AGE_SHIFT`/`GK_AGE_SHIFT` were tuned to close (a 40-season dynasty audit caught it immediately: mean climbed to ~74.5 with 90+ at 17.7% of the AI pool by season 25, instead of staying near 0%). Confining the bias to growth years, plus a magnitude pass (`PROGRESSION_BIAS_SD_YOUNG` 3→1.5, `PROGRESSION_FORM_SD_YOUNG` 5→3) to also reduce the previous first-pass extremes (a -10-in-one-season swing at age 22 was the specific user complaint), fixed it.

Declines are never damped — a bust should decline just as easily at 75 ovr as at 55; the resistance is specifically to getting *better*, not to getting worse. Verified (3000-sample check, fixed age 22): mean/max positive delta shrinks from +1.48/max+9 at 55 ovr, to +1.06/max+7 at 72 ovr, to +0.11/max+5 at 85 ovr — a real soft ceiling on top of the age curve. **AI formation selection:** every AI club automatically lines up in the shape that fields its strongest XI. `chooseBestFormation(roster)` (`formations.ts`) runs `selectXI` for each formation and returns the one whose auto-picked XI has the highest combined OVR (ties → 4-3-3; a roster of ≤11 fields the same set in every shape so all tie — the choice only bites once a squad is deep enough that different shapes start different players, e.g. a striker-rich squad favors a two-forward shape). `assignAIFormations(teams, players, userTid)` (`src/core/teams/clubs.ts`) applies it to every club **except the user's** (whose choice is never overwritten), called at league creation (`createLeagueState`) and once per offseason (`simOffseason` step 6.6, on the settled rosters). **Known granularity limit:** an AI club's formation is refreshed only at the offseason boundary, so a mid-season winter transfer doesn't re-trigger it until the next offseason — consistent with the game's other offseason-granularity AI decisions.

A post-review hardening batch (bug-hunt sweep, 2026-07-11) fixed several correctness issues without changing any game design: all league-mutating UI actions are now serialized through a single promise chain in `LeagueContext` (previously two quick actions could compute from the same stale snapshot and silently lose the first write — e.g. sign two free agents fast and one vanished while both budgets were charged), the sim overlay now persists its result *before* closing, manual releases enforce the same per-position depth floor the transfer market's `isForSale` uses (shared `keepsDepthFloor` in `src/core/freeAgency.ts` — previously a user could release their entire roster, which crashed the match engine and bricked the save), GK↔outfield lineup swaps are rejected at both the UI and the action layer, the sim worker rejects its promise on a crash instead of hanging the "Simming" overlay forever, lingering injuries are healed at the offseason rollover, and `migrateLeague` tolerates pre-M3 saves whose played matches have no `boxScore`.

Multi-league support (BBGM-style) was added after M6: the IndexedDB `leagues` store already supported multiple saves keyed by `lid` (`src/db/leagueDb.ts`), but the UI/state layer previously auto-loaded whichever save was first in the store. New leagues are now named after the chosen club (previously every save was hardcoded to "My League", which was fine for one save but indistinguishable once multiple exist).

**Club identities are fictional and user-customizable** (2026-07-12): the default `CLUBS` list (`src/core/teams/clubs.ts`) was switched from real Premier League clubs to 20 invented English-flavored clubs (trademark caution — real club names are PL-licensed marks; deliberately fictional rather than PES-style soundalikes, per user decision), and a "Customize Teams" button on each save in `/leagues` opens an editor to rename any club and change its abbreviation/colors per save (`customizeTeamsAction` in `LeagueContext` + pure `applyTeamIdentities` in `src/core/teams/customize.ts`; renaming the user's own club also renames the league while the two names still match). ## Open design decisions (from the 2026-07-11 bug hunt)

A full-codebase review (PR #34 fixed the mechanical bugs) left four findings that need a **design call with the user before coding** — each has more than one reasonable fix. Whoever picks these up: discuss options first, don't just implement.

1. **Winter-sale wage dodge.** Season wages are charged once, at offseason settlement, against whoever holds the player *at that moment* (`wageBill` over the current roster in `simOffseason` step 3.5). Selling a high earner in the winter window therefore dodges his entire season wage bill; the buyer — who had him for half a season — pays all of it. Exploitable by the user. Options: pro-rate wages by ownership stints, charge the seller their share at transfer execution, or accept it as a deliberate quirk and document it.
2. **The "no club ever runs a deficit" tuning breaks at the 30-man cap.** It was only ever proven for the 25-man `ROSTER_COMPOSITION`: worst case 30 players × `SALARY_PER_OVR` (20k) × 99 OVR = $59.4M wages + `SCOUTING_SPEND_MAX` ($20M) = $79.4M > `BASE_SEASON_BUDGET` ($75M), so a stacked full user squad with a low finish goes negative. Options: retune the budget constants for 30, cap scouting spend relative to the wage bill, or decide user deficits are actually allowed (the invariant was originally about AI clubs) and add debt UI/consequences.
3. **Youth intake positions don't match roster needs.** `generateYouthIntake` draws positions uniformly (12.5% each) while rosters need CB/FB/CM at 16% each, so over long dynasties the league drifts structurally short of those positions and `runAIFreeAgency` silently gives up when a position's free-agent pool is empty. Likely fix: weight the youth position draw by `ROSTER_COMPOSITION` — but confirm the change to rng-stream consumption is acceptable (it shifts every downstream roll in that offseason).
4. **Mid-season transfer stat attribution.** `SeasonStats` is one running total per player per season with no per-club stint, and Leaders/Roster attribute the whole line to the player's *current* club — buy a 10-goal striker at the winter deadline and your team gets display credit for all 10. Options: per-stint stats (schema change + save migration) or keep the single total and add a "joined mid-season" marker in the UI.
5. ~~Division 2 long-dynasty OVR drift~~ — **resolved**, see the "Second division" section's "Rebuilt from first principles (2026-07-15)" entry below: the probabilistic `wouldRefuseExtension`/free-agency-priority approach this item originally described was replaced by a deterministic `enforceDivision2Ceiling` sweep, which a 30-season × 3-seed dynasty audit confirmed holds "D2 strongest ≤ D1 average" in all 3 seeds with Division 2's max OVR flat across the dynasty (essentially zero compounding drift) — no longer an open design question.

Lower-severity leftovers from the same review, no design input needed, just not yet done: no guard anywhere against fielding fewer than 11 when a roster drops below 11 healthy players (`selectXI` silently skips slots — needs a rule: emergency call-up, forfeit, or hard block) — **partially addressed for the user's own club** by the Academy feature's `ensureUserRosterSafety` (see below), which auto-promotes from the user's academy if their senior roster falls below `ROSTER_SAFETY_FLOOR`; AI clubs and any other path to a sub-11 roster are still unguarded. `pickReplacement` in `matchSim.ts` would hand an outfielder back as a "GK replacement" if GKs could ever be subbed/injured/carded (currently unreachable — a landmine for whoever adds GK injuries).

A FotMob-style **Match Rating** stat was added post-M6. `computeMatchRating()` (`src/engine/matchRating.ts`) scores each player's `PlayerMatchLine` 0.0–10.0 from a fixed 6.0 baseline using a position-weighted matrix (GK/DEF/MID/FWD columns for goals, assists, shots on target, tackles, saves, clean sheets, goals conceded, cards) supplied by the user, applied only to the stats this engine's box score actually tracks — pass accuracy, big-chances-created, own goals, errors-leading-to-goal, and a separate penalty-save flag aren't modeled since the sim has no underlying data for them. A player's swing away from 6.0 is damped by `minutesPlayed / 90` so cameo subs can't post extreme ratings off a couple of touches. `minutesPlayed` itself is new too: reconstructed from each match's substitution/red-card/injury event clocks in `simMatchDetailed` (`src/engine/matchSim.ts`), not tracked directly during the sim loop. Rating and minutes are stored per-line on `PlayerMatchLine` (shown on the Box Score page) and rolled up into a new `SeasonStats.avgRating` (via `ratingSum`), surfaced as a sortable column on the Stat Leaders page. `src/db/migrate.ts` backfills both fields for pre-existing saves — historical box scores get a neutral 0 minutes / 6.0 rating since the clock data needed to reconstruct them doesn't survive in old saves.

A **Finance** page (`src/ui/pages/Finance.tsx`, sidebar under Team, linked from the Dashboard's Finances card) was added post-M6 to surface the existing M6 finance model in one place: current budget/hype plus the scouting-spend slider (same control as the Dashboard's), a season-settlement breakdown mirroring `runOffseason`'s math exactly (base + prize tier by rank + damped hype revenue − wage bill − scouting spend; labeled a projection during the regular season, the final settlement during the offseason), the full per-player wage-bill table, the user's transfer history with all-time spend totals, and a league-wide budget/hype/wage-bill table. Beyond an `ordinal()` helper moved into `src/ui/format.ts`, the page itself is pure display; the Dashboard's Finances card also gained a "Wages due at season end" line.

**Wages were reworked in the same PR** (2026-07-11): the flat 20k-per-ovr placeholder salary became a Premier-League-calibrated cubic — weekly wage = `WAGE_WEEKLY_MIN + WAGE_WEEKLY_COEFF × (ovr − WAGE_OVR_FLOOR)³` times a deterministic per-signing ±15% roll (`seasonSalaryForOvr` in `src/core/contracts.ts`, used by contract terms, AI free agency, and player generation). ~£22k/wk at 60 ovr → ~£314k/wk at 90, versus the old 23k→34k range. `BASE_SEASON_BUDGET` rose 75M→95M to keep AI clubs deficit-free: the old "base covers 25×99-ovr wages" invariant is unreachable under the cubic, so the tested guarantee is now "base covers `WAGE_SAFE_SQUAD`", a benchmark shaped from 25-season dynasty audits (max observed AI wage bill ~86M, settlement margins never below +32M). A user hoarding an elite ROSTER_CAP squad *can* now out-wage the base — the Finance page projects the shortfall (documented gap, sharper than before). Old saves keep their existing cheap contracts until players re-sign at the new rates. **Wage timing also changed**: wages are now paid up front at each season's start (`chargeSeasonStart` — base allocation in, squad wages out, applied at league creation and in `simOffseason` on the finalized new-season rosters), while season-end settlement (`settleSeasonEnd`) handles only prize + hype − scouting. A new club therefore starts with `BASE_SEASON_BUDGET − its initial wage bill` (~50-65M) instead of a full 95M, in-season cash is genuinely spendable, and mid-season acquisitions (transfer buys / free-agent signings during the regular phase) charge the player's full season salary at acquisition on top of any fee.

The `tackles` catch-all-volume issue flagged above was fixed post-M6: every simulated turnover now rolls a three-way credit outcome (`TACKLE_CREDIT_PROB`/`INTERCEPTION_CREDIT_PROB` in `src/engine/constants.ts`) instead of unconditionally logging a tackle, splitting the old single stat into `tackles` and a new `interceptions` (`PlayerMatchLine`/`SeasonStats`, migrated to 0 for old saves). Interception credit is weighted by each player's previously-unused `interceptions` skill rating via a new `pickInterceptor` (`src/engine/attribution.ts`), rather than reusing `tackling` for both stats. `matchRating.ts` gained a matching `INTERCEPTION_WEIGHT` position matrix. Match flow/possession/scoring are untouched — only what gets credited as a stat changed. Both new UI columns (Box Score, Stat Leaders, Roster) and the Manual were updated to match. Design doc: `docs/superpowers/specs/2026-07-14-tackle-interception-split-design.md`.

**`BASE_SEASON_BUDGET` was cut 95M→50M (2026-07-13)**, per user report that a bottom-table ("poverty") club had banked $163M+ after a few frugal seasons — investigation found no per-club budget variance or rank-payout bug; `budget` is by design a cumulative running balance that's never reset to the base allocation each season (only added to via `chargeSeasonStart`/`settleSeasonEnd`), so light spending simply compounds indefinitely. `WAGE_WEEKLY_MIN`/`WAGE_WEEKLY_COEFF` were rescaled in tandem (2,000→1,000 / 2.5→1.3, ~52% of the old cubic) to hold the `BASE_SEASON_BUDGET > WAGE_SAFE_SQUAD` solvency invariant with a comparable margin — new wage table: ovr 60 ≈ 11.4k/wk → 90 ≈ 163.5k/wk (was 22k→314k). `HYPE_REVENUE_PER_POINT` also rose 500k→750k per user request to make hype revenue "a little bigger" relative to the smaller base (max hype revenue 20M→30M).

**`BASE_SEASON_BUDGET` raised again, 50M→110M (2026-07-14, PR #63)**, alongside the second generation retune above (M1 section) — raising league-wide ovr raises `WAGE_WEEKLY_COEFF`-driven wages cubically (`WAGE_WEEKLY_MIN`/`WAGE_WEEKLY_COEFF` themselves were untouched this time), so the 50M base immediately broke AI solvency once generation actually reached the higher ovr range. `WAGE_SAFE_SQUAD`'s benchmark squad shape was retuned from `[11,73],[7,66],[7,56]` to `[11,80],[7,74],[7,65]` to match. Re-audited clean (20-season × 3-seed dynasty run: min AI budget +£32M, was -£26M under the old base against the new ovr distribution). A follow-up code review flagged that `BASE_SEASON_BUDGET` is now over a third of `MAX_BUDGET` (300M), so AI clubs saturate at the cap much faster (empirically: mean AI budget already ~$182M by season 2) — not yet fixed, since it doesn't break the solvency invariant, but it does erode the relative-wealth signal `clubContext.ts`'s frugality/ambition scalars and the AI market's reserve-fraction differentiation depend on. Worth a fresh look if AI club behavior starts feeling homogeneous.

**A hard `MAX_BUDGET` cap followed immediately after** (same day), since the base cut alone only slowed the compounding rather than bounding it (a first audit still showed four-figure-million budgets by late dynasty). `clampBudget()` (`src/core/finance/budget.ts`) is applied at every site a club's budget can increase: `chargeSeasonStart`, `settleSeasonEnd`, `executeTransfer` (the shared user↔AI transfer function in `transfers/negotiation.ts`, covering both outgoing transfers and accepted inbound offers), and the AI↔AI market's seller-side credit in `ai/transferMarket.ts`. Spending is never restricted by the cap — only banking past it is. Initially set to $200M, then bumped to **$300M** the same day: `trueTransferValue`'s base curve alone prices a 90-ovr player at ~$201M before age/potential/contract multipliers stack on top, so $200M would have sat right on top of the valuation ceiling and made the league's most elite players structurally unaffordable for every club, not just rare big-money buys. Re-audited via a 20-season × 3-seed dynasty run (real `simThrough`/`simOffseason`): zero AI deficits, budgets cleanly saturate at exactly the cap with no team ever observed over it. All 328 existing tests pass unchanged.

**Finance/market tuning (2026-07-18):** two shipped changes, from a user report that AI GMs sit on huge idle balances plus a want for elite players to be effectively untradeable. Audited together (2-seed, 30-season):
- **`MAX_BUDGET` is now a hype-scaled ceiling, not a flat cap** — reduces cash-hoarding. Raised $300M→$400M first (per user ask), then made $400M the *top* of a fame scale: `budgetCap(tier, hype)` (`src/core/finance/budget.ts`) interpolates between `MAX_BUDGET_FLOOR` ($200M) at hype 0 and `MAX_BUDGET` ($400M) at `HYPE_MAX`, times `DIVISION_2_BUDGET_SCALE` for tier 2. `clampBudget` and `chargeSeasonStart` gained a `hype` param, threaded through all ~9 call sites (season settlement, transfer/loan/cup-prize credits, the AI↔AI market + division-ceiling seller credits — the latter two via a new per-tid hype map). Old saves migrate with `t.hype ?? HYPE_INITIAL`. Audited effect: mean AI budget ~150M→~118M, median ~122M→~100M; solvency holds (min ~£56-60M — the cap only bounds *banking*, never causes a deficit) and OVR equilibrium is intact. **Honest caveat:** it does *not* produce cash-rich "big clubs" as first pitched — famous clubs *spend* their higher allowance rather than bank it, so their cash balances aren't higher; the real effect is lowering the hoarding ceiling for weaker clubs.
- **"Priceless star" elite valuation premium** (`VALUATION_ELITE_THRESHOLD`/`_COEFF`/`_EXPONENT`, `src/core/finance/valuation.ts`) — the base value curve is flat at the top (90 ovr ≈ $201M), so AI clubs freely traded even the very best players. A steep premium above ovr 85 (`+COEFF × (ovr−85)^EXPONENT`, added to the base) rockets a generational talent's asking price past the $400M cap so no club can afford him: 85 ≈ $156M, 88 ≈ $244M, 90 ≈ $425M, 92 ≈ $739M. Tuned "only the very top" (with the user) so 85-88 stars stay expensive-but-payable and only 90+ becomes unsellable. Audited: **0 permanent transfers ≥$300M over 30 seasons** (90+ never move) vs ~46-48 near-elite ($200-300M) deals; mid-market churn unaffected (~1230 transfers/season, 11-19 distinct D1 champions/30). This also fixed a "club renews a star then the market immediately sells him" oddity (an unaffordable player simply can't be moved).
- **Tried and reverted:** `AI_MARKET_MAX_BUYS`/`MAX_SELLS` 3→5 (to spend down hoarded cash faster) — it did cut hoarding further and raise volume ~+39%, but made the market aggressive enough to trade *just-renewed* stars and destroyed a lot of money via cap-clamping on the sell side (~977M/window in a fresh-league test, blowing the money-conservation regression guard). Reverted to 3 once the elite premium solved the star-churn concern more cleanly.
- Note this landed in the same conversation as (and after the user rejected) a larger "persistent league hierarchy" idea — a 30-season audit found near-total promotion/relegation churn (all 40 English clubs cycle through Division 2; ~87 promotions), and hype-revenue-boost / fewer-swaps / academyBase-persistence prototypes were all tried and **discarded** when the user decided the churn was actually desirable.

A multi-phase effort to replace the AI's rule-scripted behavior with a Basketball-GM-style evaluation system, where a club's strategic direction and how it values any given player both *emerge* from its current state rather than from hand-authored per-club scripts (full brief: the user's "AI General Manager Philosophy" doc — ask for it). Planned spine: (1) evaluation core, (2) AI↔AI transfer market, (3) AI inbound offers for the user's players, (4) proactive contract renewals, (5) imperfect/scouting-noised decisions. (Phase 4 was originally scoped as "contract renewals, sell-at-peak, replace-aging"; sell-at-peak turned out to already exist as an emergent effect of phase 2's transfer market, and replace-aging turned out not to need a separate mechanic — see the phase-4 bullet below.)

- **Phase 1 — evaluation core: done (this PR), no behavior change yet.** New `src/core/ai/` module, not yet consumed anywhere in the sim (so no observable change; expect the constants to be retuned once phases 2+ actually run on top of it). Two pieces:
  - `clubContext.ts` — `deriveLeagueContexts(league)` computes, for every club, a `ClubContext` derived entirely from current state: squad strength (mean ovr of best `AI_SQUAD_STRENGTH_COUNT`), mean age, per-position depth/best-ovr, and two league-normalized scalars — **ambition** (win-now pressure: weighted blend of wealth/fame/squad-strength/form) and **frugality** (financial caution: inverse of relative wealth) — plus a display-only `StrategicDirection` label. Per the user's call, philosophy is *derived from state*, not stored, so there's no schema change and no save migration.
  - `evaluate.ts` — `valueToClub(player, ctx)` / `evaluatePlayerForClub(...)` layer three club-relative multipliers on top of the existing market value (`trueTransferValue`): **positional need** (depth vs `ROSTER_COMPOSITION` × upgrade over the incumbent), **timeline** (age × ambition — win-now clubs pay up for prime, developer clubs for youth), and **affordability** (deal cost vs budget, penalty scaled by frugality). Returns a full breakdown for a future "why did they value him at X" UI. This is the primitive phases 2-5 build on (buy if value ≥ asking, sell if value < market, etc.). All tuning constants live in the `AI evaluation core` section of `src/core/constants.ts`.

- **Phase 2 — AI↔AI transfer market: done.** `src/core/ai/transferMarket.ts`'s `runAITransferMarket(...)` has AI clubs trade with each other, driven entirely by phase-1 `valueToClub` (the user's club is excluded from both sides — inbound offers for the user's players are phase 3). The whole market is one comparison applied everywhere: **a player's reservation price is his value to his *current* club, and he moves to whichever other club values him enough more to clear that reservation and afford the fee** (fee lands `AI_MARKET_FEE_SHARE` of the way between the two valuations; money conserved). Surplus dumping, sell-at-peak (an aging star's keep-value drops below his market price → available), and needs-based buying all emerge from that single rule — no scripted "if rebuilding…" branches. Guardrails: a club only shops players it values at ≤ `AI_MARKET_AVAILABILITY` × market (won't auction an irreplaceable core player), buyers need a `AI_MARKET_MIN_SURPLUS` margin to bother, per-club `AI_MARKET_MAX_BUYS`/`MAX_SELLS` caps per window, live enforcement of the depth floor (`keepsDepthFloor`), `ROSTER_CAP`, and affordability. Clubs hold back a **cash reserve** rather than blowing their whole budget: a buyer spends only the surplus above `reserveFraction × live budget`, where the fraction runs from `AI_MARKET_RESERVE_FRACTION_MIN` (wealthiest/least-frugal clubs) up to `AI_MARKET_RESERVE_FRACTION_MAX` (poorest), so no club is ever spent down to £0 (dynasty-audited min budget ~£0.3M+ over 120 season-years, vs a literal £0 before the reserve). Deterministic: valuations snapshotted at window open, deals executed in descending-surplus order, only randomness a seeded per-buyer valuation jitter (`AI_MARKET_VALUE_JITTER`, a first taste of phase-5 scouting imperfection). Tuning constants in the `AI↔AI transfer market` section of `constants.ts`.
  - **Runs once per window, both windows.** Summer: a new step in `simOffseason` (after `trimRosterSurplus`, before season-start wages), valued for the upcoming season with the just-finished season's form. Winter: fired inside `simThrough`'s matchday loop the first time a batch reaches `WINTER_WINDOW_OPEN_MATCHDAY` (18), which is why `simThrough` now threads team/budget/transfer state through the loop instead of snapshotting `league.teams` once. A new `LeagueStore.winterMarketRunSeason` (migrated to `null`) guards the winter market to exactly once per season regardless of how the user batches matchdays.
  - **Verified** via a 40-season × 3-seed dynasty audit (real `simThrough`/`simOffseason`): no AI club ever goes into deficit, OVR stays at its prior equilibrium (mean ~58, 80+ <1%, 90+ 0% — the market causes no inflation), AI rosters stay ≤ `ROSTER_CAP`, and clubs make ~103 transfers/season league-wide. Two existing tests were re-pointed at the corrected invariants (AI clubs now spend, so budgets no longer only grow — the real invariants are AI solvency + a rising league-wide total; AI squads can now carry up to `ROSTER_CAP`, not just the 25-man composition). A **News Feed** page (`src/ui/pages/NewsFeed.tsx`, `/news`, sidebar under League) surfaces the whole league's transfer activity — every completed deal in `league.transfers` (AI↔AI included), newest first, grouped into per-season cards with window/player/from→to/fee columns, club/season filter dropdowns, and user-club deals row-highlighted. Pure display over existing state (no schema change).

- **Phase 3 — AI inbound offers for the user's players: done.** `src/core/transfers/inboundOffers.ts` mirrors phase 2's evaluation-driven design but with the user as the seller and full counter-negotiation (a design call made with the user: offer timing recomputed per-window like Recommended Transfers rather than fired once at window open; negotiation goes both ways rather than take-it-or-leave-it; own `/incoming-offers` page rather than a section on Transfers).
  - `inboundOfferCandidates(league)` scans the user's roster and, for each player, finds the single AI club that values him most via the same `valueToClub`/`ClubContext` primitive phase 2 uses, offering only if that club clears the player's worth to the user's *own* club (`AI_MARKET_MIN_SURPLUS`, reused) — capped to `INBOUND_OFFERS_MAX` most compelling offers, with the same fee-split (`AI_MARKET_FEE_SHARE`) and a per-player seeded jitter (`INBOUND_OFFER_VALUE_JITTER`) for opening-offer variety. A buyer's ceiling is capped by what it can actually spend without dipping into its `AI_MARKET_RESERVE_FRACTION_*` cash reserve (`buyerSpendable`) — otherwise a club with a high-but-unspendable valuation would dangle offers it could never actually close.
  - **Negotiation is symmetric to outgoing offers but mirrored**: `respondToAsk(ceiling, ask, priorAsks)` is the buyer's-side twin of `negotiation.ts`'s `respondToOffer` (every "higher is better" comparison flipped to "lower is better"), reusing the exact same `NEGOTIATION_LOWBALL_FACTOR`/`COUNTER_PADDING_*`/`NEGOTIATION_MAX_ROUNDS` tuning so both directions of the market haggle at the same pace. `acceptInboundOffer`/`rejectInboundOffer`/`counterInboundOffer` persist state in a new `LeagueStore.inboundOffers` array (migrated to `[]` for old saves), separate from the outgoing `negotiations` array since the user is the seller here, not the buyer. `executeTransfer` (previously private to `negotiation.ts`) is now exported and shared by both directions.
  - **UI**: a new `/incoming-offers` page (sidebar under Team, next to Transfers) lists live offers with Accept/Counter/Reject controls, an "Other Offers" section for negotiations that outlived the current candidate list (mirrors Transfers' "Other Negotiations"), and a "Sold This Window" section — an accepted offer sells the player immediately, so without this he'd just silently vanish from the list with no on-page confirmation.
  - Browser-verified end to end (reject, counter → AI raises its offer → accept, and immediate accept) via a real dynasty run to the winter window.
  - **"List for Transfer" (post-phase-3 addition)**: a new `StoredTeam.transferListed: number[]` (migrated to `[]` for old saves) lets the user explicitly flag their own roster players as available, via a button in the Roster page's pitch-chip popover and bench table. `inboundOfferCandidates` reads it to lower the surplus bar a buyer needs to clear from `AI_MARKET_MIN_SURPLUS` to a much smaller `LISTED_FOR_TRANSFER_MIN_SURPLUS`, and to prioritize listed players ahead of unlisted ones within the `INBOUND_OFFERS_MAX` cap — not a guaranteed offer, just meaningfully better odds, consistent with "listing signals willingness, doesn't force a sale." `setTransferListed(league, pid, listed)` (`inboundOffers.ts`) is a no-op for a pid not on the user's own roster.

- **Phase 4 — proactive contract renewals: done.** `src/core/ai/renewals.ts`'s `runAIContractRenewals(...)` closes a gap phases 1-3 didn't touch: previously a contract just expired (`releaseExpiredContracts`) and the player hit the open free-agent pool with zero priority for his own club to keep him — `runAIFreeAgency` only reacts to positional shortfalls, not "this is my good player." Now, as a new step 0 in `simOffseason` (before contracts expire and before AI free agency/the summer transfer market run), any AI player entering his contract's final season is renewed on the spot if `valueToClub(player, ctx) ≥ contractTerms(player, nextSeason).salary × AI_RENEWAL_MARGIN` — reusing the phase-1 evaluation core and the same one-button `contractTerms`/`extendContract` helpers the user's own renew button already calls, so no new pricing or contract-length logic was needed. `valueToClub` already folds in affordability, so a cash-strapped club naturally lets pricier players walk without a second budget gate. The original phase-4 scope also named "sell-at-peak" and "replace-aging," but investigation found both already covered: sell-at-peak is an existing emergent effect of phase 2's transfer market (a keep-value that drops below market price makes a player available), and replace-aging needs no separate mechanic — an old, declining player simply fails the renewal bar, walks (or is sold earlier via the existing market), and the vacancy is filled by the same free-agency/transfer-market machinery that fills any other hole. Verified via a 40-season × 3-seed dynasty audit (real `simThrough`/`simOffseason`): zero AI deficits, ~7,400 renewals/seed over 40 seasons (meaningfully active, not inert), and the league-wide OVR equilibrium unchanged from the pre-phase-4 baseline (80+ under 0.2%, 90+ at 0%). Design doc: `docs/superpowers/specs/2026-07-12-ai-gm-phase4-contract-renewals-design.md`.

- **Phase 5 — imperfect/scouting-noised decisions: done.** Every AI valuation that decides a transfer, an inbound offer, or a renewal now runs through a new `perceivedValueToClub(player, ctx, jitter)` (`src/core/ai/evaluate.ts`) instead of the raw `valueToClub` — a deterministic ± jitter scaled by `scoutNoiseFraction(ctx)`, which reuses the phase-1 `frugality` scalar as a stand-in for scouting investment (no new `ClubContext` field, keeping the "derived not stored" philosophy): the wealthiest club in the league (frugality 0) judges players almost exactly right (± `AI_SCOUT_NOISE_MIN`, 2%), the poorest (frugality 1) can be off by up to `AI_SCOUT_NOISE_MAX` (8%). This replaces the old flat `AI_MARKET_VALUE_JITTER`/`INBOUND_OFFER_VALUE_JITTER` (both a uniform 0.04 for every club regardless of wealth, called out in their own comments as "a first taste of phase-5 scouting imperfection" when phases 2-3 shipped) with one shared noise model applied consistently in three places: the AI↔AI market's buyer valuation (phase 2, `transferMarket.ts`), an inbound buyer's opening-offer ceiling (phase 3, `inboundOffers.ts`, unchanged: still no noise on the live re-evaluation at counter time, only opening-offer variety), and — new — a club's renewal decision about its own player (phase 4, `renewals.ts`, previously fully deterministic with zero rng; now seeded per-player off `hashInts(seed, pid)` so a poorly-scouted club can misjudge a player it already owns, not just one it's buying). Seller-side reservation prices in the AI↔AI market stay unnoised by design (out of scope — the market's fee-floor math wasn't touched). Verified via a 15-season dynasty audit (real `simThrough`/`simOffseason`): no AI club ever went into deficit (min budget ~£38.8M) and the OVR equilibrium held (mean ~57, 80+ at 0%), i.e. the noise perturbs individual deals without disturbing the league-wide invariants phases 1-4 already established.

**Refinements deferred:** the winter AI↔AI market fires once at window open rather than trading dynamically across the window; phase 3 offers are recomputed fresh each render rather than snapshotted at window open, so a candidate can in principle drift or disappear between viewings if the underlying club state changes (mirrors how Recommended Transfers already behaves, not new to phase 3); phase-4 renewals are decided once, at the offseason boundary, using state as of that moment — there's no in-season contract-year tracking to revisit a decision mid-season (matches the game's existing offseason-only contract granularity, not a new limitation); phase-5 noise only covers AI valuations of players — the user's own valuation noise (Recommended Transfers, Finance projections) is the separate, pre-existing M6 scouting-spend mechanic and wasn't touched.

## End-of-season awards

Player of the Season, Golden Boot, and a Team of the Season pitch view were added, per the user's "Prompts" Google Doc ("End of season awards") plus a user-requested Golden Boot addition. New `src/core/awards.ts` (`computeSeasonAwards(players, season)`), pure and re-runnable for any past season since `Player.stats[]` is append-only and never pruned.

- **Scoring, not just a leaderboard read.** Per the user's explicit ask, both individual awards layer explicit stat weights on top of `avgRating` (itself already a position-weighted blend of the same stats via `computeMatchRating` in `matchRating.ts`) rather than just sorting by rating: **Player of the Season** (`potyScore`) adds `avgRating + goals × POTY_GOAL_WEIGHT[group] + assists × POTY_ASSIST_WEIGHT[group]`, weighted noticeably heavier than match rating alone already credits goals/assists — a browser-verified run had the POTY winner (17G/12A, 6.81 avg rating) over higher-pure-rating alternatives. **Golden Boot** is a simple most-goals-in-the-season lookup (tie-break: fewer appearances, then more assists, then pid), no formula needed. **Team of the Season** (`totsScore`) goes further per the user's ask to weigh goals/assists/tackles/interceptions/saves, not just rating: `avgRating` plus position-weighted goals, assists, tackles, interceptions, GK saves, and a goals-against penalty — one player per `FORMATIONS["4-3-3"]` slot, highest `totsScore` at that exact position, no player reused across slots. All weight constants live in the "End-of-season awards" section of `src/core/constants.ts`.
- **Eligibility**: `AWARD_MIN_APPEARANCES` (15 of 38 matchdays) gates Player of the Season and preferentially gates Team of the Season slots (a slot only falls back to an under-threshold player if no one at that position qualifies at all, so every slot still fills even at a thin position). Golden Boot has no minimum — goals scored is unambiguous regardless of appearances.
- **Storage**: a new `awards: SeasonAwards` field on `SeasonHistoryEntry` (`src/core/standings.ts`), computed in `simOffseason` from `league.players` (not the offseason's mutated `players` variable — a retiring player still gets credit for the season he just finished, before he's filtered out of the roster) and appended alongside the existing table/teamStats snapshot. Old saves are migrated in `src/db/migrate.ts` by actually recomputing `computeSeasonAwards` for every historical season, rather than defaulting to nulls — unlike `teamStats`, past awards are fully reconstructable since per-season player stats are never pruned.
- **UI**: a new `/awards` page (sidebar under League, next to Stat Leaders) with a season dropdown, two award cards, and a read-only Team of the Season pitch view reusing `layoutSlots`/the `.pitch-field`/`.pitch-chip` CSS from the Roster page's `PitchField.tsx` (a separate lightweight render, not `PitchField` itself, since that component is coupled to drag/drop and release/extend actions the Awards page doesn't need). Per the doc's ask that awards "be automatically displayed once the user is done simming a season," the Dashboard's "Advance to [next season]" button now navigates to `/awards` as soon as `offseasonAction()` resolves.
- **OVR-quality term added (2026-07-14, PR #63)**, after a user report that Team of the Season was picking 50-ovr players: `potyScore`/`totsScore` were built entirely from in-match performance stats with no sense of how good a player actually is, so a mediocre player on a struggling team could rack up huge tackle/interception counts just from facing more attacks and out-score a genuinely elite player with a quieter statistical season. Both formulas now add `(ovr − AWARD_OVR_BASELINE) × AWARD_OVR_WEIGHT` (`ovrBonus` in `awards.ts`), baseline 65 ("average starter"). This surfaced two follow-up bugs, both fixed in the same PR: the term originally read `Player.ovr` (present-day, mutable) directly, which corrupted `migrate.ts`'s historical-award backfill (recomputes awards for arbitrary past seasons from today's player pool) — fixed via `ovrDuringSeason()`, which reads the correct value from the already-stored `Player.hist` snapshots instead. And `AWARD_OVR_WEIGHT` was initially tuned (0.15) against a league that, at the time, still generated 7-10 points below its OVR targets (see the M1 section's second retune above), so it was never exercised against real elite players — once generation was fixed to actually reach 80-90, the same weight made a 90-ovr player's bonus alone worth ~47 goals equivalent, turning awards into a near-pure ovr leaderboard. Retuned to 0.06; verified against real simulated seasons that Team of the Season selections span a believable ovr range (60s-80s) rather than clustering at the top regardless of stats.

## Youth Academy (holding pool, replacing straight-to-roster youth intake for the user)

Per the user's "Prompts" Google Doc ("Academy") — design options were sketched ahead of time in a separate Google Doc, "Hard Feature Design Notes (Claude brainstorm)" (id findable via Drive search `title contains 'Hard Feature'`), and several of its open questions (Incoming Talent's scope, academy wages, the forced-review gate, AI academy handling) were resolved with the user before implementation. Only the user's own club gets a real academy holding pool; AI clubs' youth intake is unchanged (straight to roster, trimmed like any other offseason surplus) — the brainstorm doc's "reuse `valueToClub`" sketch for AI academy evaluation turned out unnecessary once this simpler equivalent was confirmed with the user.

- **`StoredTeam.academyRoster: number[]`** (`src/core/teams/clubs.ts`), parallel to `roster`, migrated to `[]` for old saves. In `simOffseason` (`src/core/offseason.ts` step 5), the user's youth intake is routed here instead of `roster` (on a flat stipend contract — see below); AI clubs' youth intake is untouched. `freeAgentPids` (`src/core/freeAgency.ts`) now excludes `academyRoster` pids too, and `releaseExpiredContracts` clears expired contracts from both arrays.
- **Flat stipend wages**: academy contracts (`academyContractTerms`/`extendAcademyContract`, `src/core/contracts.ts`) pay a cheap flat `ACADEMY_STIPEND_WEEKLY` regardless of ovr, not the normal ovr-cubic formula (`seasonSalaryForOvr`) — an unpromoted prospect isn't yet competing for a senior wage. Promoting a player re-contracts him at the normal formula (`contractTerms`), same wage-timing convention as `signFreeAgent` (mid-season promotions charge the season's wage immediately; offseason promotions are covered by the next season-start charge).
- **Core actions** (`src/core/freeAgency.ts`): `signToAcademy` (a free agent → a team's academy, gated by its own `ACADEMY_ROSTER_CAP`, separate from the senior roster's `ROSTER_CAP`), `promoteFromAcademy` (academy → senior roster, gated by `ROSTER_CAP`), `releaseAcademyPlayer` (cut an academy player — no depth floor, unlike `releasePlayer`, since the academy has no fixed composition target to protect).
- **Emergency call-up safety net (`ensureUserRosterSafety`)**: a 40-season-scale dynasty audit surfaced a real crash risk while landing this feature — the user's senior roster was *already* the only roster in the game with no automatic top-up (free agency/youth intake/trim all skip the user by design so they manage manually), and was already trending toward dangerously thin from unmanaged contract expiry alone (one seeded 5-season chain dropped a fresh 25-man squad to 13 with zero manual action) even *before* this change. Diverting youth intake away from the senior roster removed the one thing that had been masking it, and the same seed crashed the match engine outright (fielding fewer than 11 — a known, previously algorithm-agnostic gap called out earlier in this file) by season 4. Fixed by promoting from the user's academy each offseason — goalkeeper first if the roster has none at all, then by ovr — until the roster reaches `ROSTER_SAFETY_FLOOR` (18) or the academy runs out; a no-op for an attentively managed roster. This is a backstop, not a normal growth path — see the "no guard against fielding fewer than 11" known-gap note above for the (still open) AI-side and non-academy equivalents.
- **UI**: a new `/academy` page (sidebar under Team) lists the user's academy with Extend/Promote/Release actions. **Incoming Talent** (`/incoming-talent`) was repurposed from a general free-agent-signing stub into a youth-only prospects page (unsigned players age ≤ `PROSPECT_AGE_MAX`, 21) with both Sign-to-Senior and Sign-to-Academy buttons; a new **Free Agents** page (`/free-agents`) took over its old general signing behavior for everyone older, unchanged otherwise.
- **Deferred**: the forced-acknowledgement gate from the user's original prompt ("add a stage of simming that requires the user to look at incoming talent") was explicitly deferred past v1 with the user — nothing currently blocks sim advancement on unviewed academy intake.

## Team OVR & POT

Per the "Prompts" Google Doc's "Team Ratings" entry ("Create a team Overall and Potential"), clubs now have an overall rating alongside the existing per-player one. ## Second division (promotion/relegation)

Per `SOCCER_GM_SPEC.md` §7's v2/M6-menu "promotion/relegation with a second division" item — design spec (`docs/superpowers/specs/2026-07-14-second-division-design.md`) and implementation plan (`docs/superpowers/plans/2026-07-14-second-division.md`) were brainstormed and approved with the user first. New leagues now generate a second, full-senior-roster 20-club division ("English Division 2") below the existing one ("English Division 1"), with real 3-up/3-down promotion/relegation each offseason — existing single-division saves are migrated to Division 1 unchanged, no retroactive split.

- **Generation**: `generateTwoDivisionLeague` (`src/core/league/generate.ts`) produces 40 teams in one rng pass — Division 1 identical byte-for-byte to the old `generateLeague` for the same seed, Division 2 generated with the same evenly-spaced strength-target formula shifted down by `DIVISION_2_OFFSET` so its strongest teams land around Division 1's average, not just below Division 1's weakest. `LeagueTeam`/`StoredTeam` gained `division: 0 | 1`; `CLUBS` grew from 20 to 40 fictional identities (indices 20-39 new, same invented-English-place-name style).
- **Promotion/relegation** (`src/core/promotion.ts`, wired into `simOffseason`): bottom 3 of Division 1's final table swap with top 3 of Division 2's, straight automatic swap, no playoff. A swapped team's `academyBase` (its generation-time strength anchor and permanent youth-intake anchor) doesn't snap to its new band instantly — `stepAcademyBaseConvergence` moves it a fraction of the remaining distance each offseason over `ACADEMY_BASE_CONVERGENCE_SEASONS` (3) seasons, so a promoted club has to earn its way up rather than get an instant boost. The user's own club is swapped by the exact same rule as any AI club — no special-casing, per the user's explicit call.
- **Finance**: Division 2 clubs' base allocation and prize tiers are scaled by `DIVISION_2_BUDGET_SCALE` (`src/core/finance/budget.ts`), reflecting the real financial gap between top-flight and second-tier football; a promoted/relegated club's budget scale switches immediately (unlike the gradual squad-strength convergence above), similar to real parachute-payment economics.
- **Two correctness gaps not called out in the original design doc, found and fixed while implementing**: `computeStandings` doesn't filter matches by the team-id list it's given, so `simOffseason`'s per-division standings calls needed `league.played` pre-filtered by division to avoid crashing on a cross-division match reference; and `deriveLeagueContexts` (the AI GM evaluation core) needed its wealth/ambition/form normalization scoped *within* each division (`src/core/ai/clubContext.ts`) — pooling both divisions would have permanently read every Division 2 club as maximally frugal, since Division 2 is deliberately poorer by design. Match composite normalization (`simThrough.ts`'s `leagueMatchData`) was similarly split per division to avoid diluting Division 1's intra-division rating spread with Division 2's weaker pool.
- **UI**: Standings, Awards, and Stat Leaders each gained a Division dropdown alongside their existing season dropdowns, scoped per-division (not combined across all 40 clubs); the News Feed needed no change since its filters are already per-club. `SeasonHistoryEntry` gained a `divisionsByTid` snapshot (a team's division *during* that season, since a later swap shouldn't relabel history) and its `awards` field became a `[SeasonAwards, SeasonAwards]` tuple.
- **Cross-division transfers work by construction**: the AI↔AI transfer market, inbound offers, and free agency all already operate over `league.teams`/the free-agent pool with no division concept, so Division 1 clubs can scout and buy Division 2 talent and vice versa with no code change beyond the per-division AI-context normalization above.
- **Dynasty-audited constant tuning** (`scripts/divisionAudit.ts`, 30 seasons × 3 seeds via real `simThrough`/`simOffseason`): `DIVISION_2_OFFSET` (7) needed no change — Division 2's strongest team lands at ~62-63 average roster OVR against Division 1's ~59-60 average, matching the "lands around Division 1's mid-table strength" design goal. `DIVISION_2_BUDGET_SCALE` was raised from an initial 0.5 to **0.6** after the first audit pass showed small AI deficits (as low as -£706k) in 2 of 3 seeds; re-audited clean afterward (min budget ~+£2.8M-£4.0M across all seeds). OVR equilibrium holds with no extra inflation from the new division (Division 1 80+ 1.2-2.0%, Division 2 80+ 0.8-1.6% — both in line with or below the pre-existing single-division baseline documented above).

**Retuned again (2026-07-15)** after a user report that Division 2's Team of the Season showed no real OVR gap against Division 1's — a 30-season dynasty audit confirmed it: the "D2's best ≈ D1's average" generation target had eroded further over a long dynasty (D2's strongest team ended up *exceeding* D1's average, and D2's TOTS came within ~2.6 OVR of D1's), driven by two effects: a too-generous generation-time offset, and `MAX_BUDGET` being a flat, undifferentiated ceiling — Division 2 clubs eventually banked as much as Division 1 clubs despite earning slower, closing the financial gap that should have let Division 1 clubs reliably outbid for Division 2 talent. Three changes:
- **`DIVISION_2_OFFSET` is now a derived formula, not a literal**: a new `DIVISION_2_TARGET_D1_RANK` picks which Division 1 finishing rank Division 2's own strongest team's generation-time target should match (`src/core/constants.ts`), self-correcting if `TEAM_STRENGTH_SPREAD`/`NUM_TEAMS` are ever retuned again (the old literal had already drifted out of sync once, per the M1 section above). Tuned to `NUM_TEAMS` — Division 1's own *weakest* team, the formula's ceiling — after empirically finding weaker ranks didn't hold.
- **`MAX_BUDGET` is now division-scaled** (`clampBudget` gained a `division` parameter, `src/core/finance/budget.ts`, threaded through every caller). `DIVISION_2_BUDGET_SCALE` was tried at the user's original ask of 0.4, found to cause real AI deficits (as low as -£20M) even worse than the 0.5 this constant was already once moved away from, and settled back at **0.6** after 0.5 also still dipped negative in some seeds — confirmed with the user that solvency took priority over the exact number.
- **New mechanic**: a Division 2 player refuses to sign a new deal once he'd already be a viable transfer target for some Division 1 club by the same `AI_MARKET_MIN_SURPLUS`/`buyerSpendable` bar Phases 2/3 already use (`wouldRefuseExtension`, `src/core/ai/breakoutRefusal.ts`) — deterministic (no jitter), computed live wherever it's checked, no schema change. Blocks both AI auto-renewal (`ai/renewals.ts`) and the user's own Extend action (`LeagueContext.tsx`'s `extendContractAction`, with matching Roster/`PitchField` UI showing "Wants a move to Division 1" instead of the Extend button); makes him immediately purchasable by bypassing the normal for-sale gate in user-initiated transfer negotiation and Recommended Transfers (`transfers/negotiation.ts`'s `isForSaleOrRefusing`, `transfers/recommendations.ts`) and in the AI↔AI market (`ai/transferMarket.ts`, bypassing its usual reservation-vs-market-price shop gate); and gives Division 1 clubs first pick of him in free agency once his contract actually lapses (`freeAgency.ts`'s `runAIFreeAgency` gained a `preferD1Pids` parameter, wired from a pre-release refusal check in `offseason.ts` — without this, plain free agency has no division concept and could just hand him straight back to another Division 2 club, since it only fills empty roster slots and was never built to let a club upgrade an existing spot). This is the one place "cross-division transfers work by construction" (above) needed an actual code change, not just AI-context normalization.
  - Verified the trigger itself fires often in real play (86% of eligible Division 2 players in one 15-season dynasty snapshot) — the earlier concern that it might be too strict (an early single-player test needed an unrealistically high 95 OVR to clear the bar) turned out to be an artifact of that specific test's club context, not the general case.
  - **Known, accepted limitation, not an oversight**: even with all of the above and `DIVISION_2_TARGET_D1_RANK` at its formula ceiling, season-1 (generation) numbers land on target (Division 2 max OVR ~80-84, TOTS avg ~65-68) but season-30 still drifts to ~74-76 TOTS avg / 85-93 max — free agency's "fill empty slots only" design means Division 1 clubs often have no open slot to actually claim a refusing star even with priority, and 30 seasons of promotion/relegation churn plus ordinary AI↔AI trading isn't fully offset by any of this. Confirmed with the user this is acceptable rather than chased further; a real structural fix (e.g. letting a Division 1 club release a weaker incumbent to make room) was explicitly scoped out as a bigger change than this pass covered.
  - A side effect of pushing `DIVISION_2_TARGET_D1_RANK` to its ceiling: Division 2's own internal quality spread widened along with the gap to Division 1 — its weakest teams can now land as low as the low-40s OVR at generation (previously bottomed out well above that), not just its strongest team moving down. Not separately audited against a target; noted here for whoever looks at this next.

**Retuned a third time (2026-07-15, same day)**, in a follow-up conversation that hadn't seen the retune immediately above land on `main` yet (two sessions working the same problem concurrently — see the "Shared repo, two Claude accounts" convention this file exists to cover). Two changes:
- **`DIVISION_2_TARGET_D1_RANK` walked back from `NUM_TEAMS` (20, D1's own weakest team) to 16.** Confirmed with the user this was the actual original intent — the design spec's own text explicitly warned against Division 2's strongest team landing "just modestly below D1's weakest team," which rank `NUM_TEAMS` did verbatim; rank 20 had been picked in the pass above as an emergency lever while fighting the dynasty-drift problem, not as a considered restatement of the target. Season-1 numbers still land close to target (D2 max OVR 81-85, TOTS avg 66.5-67.9), and all `generate.test.ts` invariants still hold (rank 16 is *further* from D1's average than rank 20 was). Trade-off found via the dynasty audit: because this narrows the starting gap, `wouldRefuseExtension`'s existing drift-fighting machinery has less room before Division 2 catches back up, so season-30 TOTS avg (74.6-78.9 across 3 seeds) is marginally worse than rank 20's documented 74-76 — an expected, not concerning, consequence of a shallower starting point, not a regression in the drift-fighting logic itself.
- **Fixed a real routing bug in the AI↔AI market that was defeating half of `wouldRefuseExtension`'s intended effect.** Diagnosed via the same dynasty audit: a diagnostic trace found that of Division 2 "refusing" players who actually got transferred in a given window, roughly half moved to *another Division 2 club*, not to Division 1 — because `runAITransferMarket`'s buyer loop has no division restriction, a refusing player being listed just makes him available to the whole market, and a rival Division 2 club can easily out-bid a Division 1 club for him (or simply be the only bidder that shows up that window), which just reshuffles talent within Division 2 instead of pulling it up and out. Fixed by restricting the buyer loop to Division 1 clubs specifically when a player's *only* reason for being listed is `wouldRefuseExtension` (a player who's normally for sale on his own merits — surplus, aging, etc. — is untouched, and can still move to any division as before). Also separately found and fixed a related RNG-stream-order bug this same investigation nearly introduced: the buyer-division filter had to be placed *after* the `perceivedValueToClub` jitter draw, not before, or skipping the draw for filtered-out buyers shifts every subsequent buyer's jitter within the same window — an initial version of this fix produced real AI deficits (min budget as low as -£5.9M in a 3-seed dynasty audit) purely from this reordering, not from the division restriction itself; correcting the draw order restored solvency (min budget +£6.0M to +£12.2M across the same 3 seeds) with no other behavior change.
  - Re-verified via the same 30-season × 3-seed dynasty audit: the "D2's strongest team vs D1's average" invariant, which was violated in all 3 seeds before this fix, now holds in 2 of 3; D2's Team-of-the-Season season-30 average sits at 73.7-76.7 (was 74.6-78.9 before this fix, similar order of magnitude). Still a **known, accepted limitation, not fully solved**: Division 2's single max-player OVR still drifts to 84-86 by season 30 against a ~70-75 target, and the free-agency `preferD1Pids` "fill empty slots only" gap noted in the pass above is untouched — this fix closed one real leak in the drift-fighting machinery, it didn't rebuild the whole mechanism.

**Rebuilt from first principles (2026-07-15, same day)**, per an explicit user mandate: "even small inflation over 30 seasons is not okay... it will compound after 100 seasons," and permission to tear down the whole drift-fighting mechanism if needed. Root-cause work (systematic-debugging style, not another patch) found the *actual* dominant driver, which every prior pass in this section had missed:

- **Diagnostic: disabled the AI↔AI transfer market entirely** (a one-line stub returning its input unchanged) and re-ran the 30-season audit with promotion/relegation still active. Division 2's max player OVR and Team-of-Season average climbed **just as much** as with the market on (e.g. TOTS 59→75, 61→77 across seeds) — proving the market was a minor contributor, not the dominant one. The real driver: **a relegated club keeps its full-strength existing roster** — only its `academyBase` (future youth anchor) converges toward Division 2's band, existing players are (deliberately, per the original design) untouched — and three clubs relegate every single season, so Division 2 permanently accumulates ex-Division-1-strength squads that take years to age out. No amount of nudging a transfer-market *chance* can fix a gap driven by players who were never up for sale in the first place.
- **This reframed `wouldRefuseExtension` itself**: investigating why the season-30 numbers were only "somewhat" bad revealed the per-club valueToClub-match version was accidentally flagging **100% of Division 2 players as refusing** (490/490 in one snapshot) — not because they were all good, but because among 20 Division 1 clubs with varied needs/ages/wealth, *some* club could always be found with a marginal fit for *anyone*. This wasn't a good sign masquerading as a bug — it was a real bug (it broke the "occasional breakout star" fiction, and would have shown the "Wants a move to Division 1" UI badge on nearly every Division 2 player). Simplified to a flat rule: `wouldRefuseExtension(player, currentTeam)` now just checks `player.ovr >= DIVISION_2_REFUSAL_OVR_THRESHOLD` (70, "a good starter" per the Manual's own scale) — no `teams`/`contexts` needed at any call site. This alone was *less* effective against the season-30 drift than the accidentally-permissive old version (fewer players qualify, so less volume drains per window) — an honest, expected trade-off, not a bug, but it clarified that **no probabilistic mechanism could ever fully solve this**, since relegation doesn't route through any market/free-agency/renewal decision at all.
- **The actual fix: `enforceDivision2Ceiling` (`src/core/ai/divisionCeiling.ts`), a guaranteed, deterministic move, not a nudge.** Every offseason, any AI-controlled Division 2 player at or above `DIVISION_2_REFUSAL_OVR_THRESHOLD` is forcibly reassigned to whichever (non-user) Division 1 club currently has the lowest average OVR at his position — recomputed after each move so multiple qualifying players spread across different needy clubs rather than piling onto one. If the receiving club is already at `ROSTER_CAP`, its own weakest player (any position) is released to free agency first, so the move always succeeds — no affordability check, no buyer-willingness check, zero fee (a structural correction, not a negotiated transfer, though still logged as a zero-fee `CompletedTransfer` for News Feed/Finance visibility). **Runs twice per offseason** in `simOffseason`: once right after promotion/relegation (before free agency, so a club that loses a player this way still gets a chance to backfill the hole — a first attempt that ran the sweep only *after* the summer market instead broke `test/validation/m4-multiseason.test.ts`'s roster-floor invariant, since a seller who lost a player there had no subsequent step to refill it), and again right after the summer AI↔AI market (before season-start wage charges) to also catch a player sold *into* Division 2 as ordinary market surplus that same offseason — otherwise he'd sit in Division 2 for a full extra season before the next cycle's sweep caught him. The function is idempotent, so the second pass is a no-op whenever the first pass already caught everything.
- **This let three now-redundant probabilistic mechanisms be removed** (torn down, not just deprecated) since the deterministic sweep supersedes them for AI clubs: the AI market's `wouldRefuseExtension`-based buyer-division restriction (`ai/transferMarket.ts`, reverted to its pre-Fix-3 form — a normal surplus sale can go to any division again, same as any other transfer, since the sweep catches an over-threshold arrival regardless), the AI renewal check that skipped extending a refusing player (`ai/renewals.ts` — renewing him first no longer blocks the sweep, since it moves players by roster membership, not contract status), and free agency's `preferD1Pids` first-pick mechanism (`freeAgency.ts`'s `runAIFreeAgency` lost the parameter entirely — a strong free agent landing anywhere is fine now, the sweep will relocate him at the next offseason if needed). `wouldRefuseExtension` itself is kept, simplified, and still used for **user-facing** purposes only: the Roster page's "Wants a move to Division 1" badge/Extend-block for the user's own Division 2 stars, and Recommended Transfers/user-initiated negotiation surfacing a rival AI club's over-threshold player as a buyable target — the user's own club is never touched by the automatic sweep, matching every other "AI clubs get automation, the user manages manually" convention in this codebase.
- **Verified via the same 30-season × 3-seed dynasty audit — this is the first pass to actually hold, not just improve**: "D2 strongest ≤ D1 average" now holds in **all 3 seeds** by a wide margin (5-6 OVR points, not a near-miss). Division 2's single max player OVR is **flat across the entire dynasty** — season 1 vs season 30: 68→69, 69→69, 68→69 (target was ~70-75) — essentially zero compounding drift, the actual thing the user asked to guarantee. Division 2's Team-of-Season average now oscillates near its ~65 target (66.5→70.3, 67.9→61.7, 66.8→66.3) instead of climbing monotonically every prior pass produced. Division 2's 80+ OVR share is 0.0% in all 3 seeds (was 0.8-5.9% across every earlier version in this section). AI solvency is, if anything, healthier than before (min budget +£43-44M across seeds, up from the +£6-12M range the probabilistic-mechanism versions produced). All 423 tests pass (two pre-existing tests needed fixture fixes unrelated to their actual intent — `test/core/offseason.test.ts`'s renewal test was accidentally picking a Division 2 club for a scenario that assumed Division 1, and the now-removed `preferD1Pids` free-agency test/renewal-refusal test were deleted since the mechanisms they tested no longer exist).

Also found: the mechanism sweeps a large fraction of Division 2 every season (~65-70 players/season, 15-20% of the whole division, since Division 1's own mean sits close enough to the 70 threshold that roughly half of any newly-relegated squad qualifies instantly). Both **not yet addressed** — deferred, not forgotten: giving the user first refusal via Recommended Transfers would mean delaying the *guaranteed* move until the summer window actually closes (there's already a `SUMMER_WINDOW_CLOSE_MATCHDAY` constant to hook into, matching the pattern the winter AI market already uses), with the deterministic sweep firing only as a backstop for whoever's left unclaimed.

A clean ablation (disabling the ceiling sweep entirely) confirmed Division 1's mean climbed to ~65-68 either way, meaningfully above the ~58-63 documented M4 equilibrium target, so this wasn't just a side effect of the sweep (though the sweep does add a further ~2-3 points on top). Root-caused and fixed via `YOUTH_BASE_OFFSET` (see the M4 section above) — this was a completely different bug from anything in this section: `generatePlayer` doesn't vary by age, so youth intake was generating 16-year-olds at full mature quality and then growing them further, inflating the league from turnover alone regardless of whether individual careers were flat. ## Loans

Per the user's ask: a player can move to another club's roster for a fixed 1-3 season commitment instead of a permanent sale, so a good player stuck behind a better one on his own depth chart can get real minutes elsewhere — minutes already feed progression (`appearances` → `minutesFactor`, see the M4 section above), so this is a genuine development lever, not just flavor. New `src/core/loans.ts`, following the same evaluation-driven pattern as the AI GM work (reuses `valueToClub`/`perceivedValueToClub`/`deriveLeagueContexts` as-is, no new AI heuristics).

- **Mechanics**: the loanee club pays a flat, non-negotiable fee up front (`computeLoanFee` = `trueTransferValue` × `LOAN_FEE_RATE` (0.08) × a diminishing per-duration multiplier — 1/1.5/1.9x for 1/2/3 seasons, calibrated off real football loan-fee research: even the priciest loans on record run far below the player's actual transfer value) and takes over his wages for the loan's length. That wage handoff needed zero special-casing — wage charging already keys off roster membership, not contract ownership, so moving the player's pid between `StoredTeam.roster` arrays (exactly like a permanent transfer) is the whole mechanism. The player's contract itself travels untouched and is unaffected by the loan.
- **User-initiated loans out**: `listPlayerForLoan` (gated by `keepsDepthFloor`, same floor as a sale/release) adds a pid+duration to a new `LeagueStore.loanListings`. AI interest is computed live and un-stored, mirroring `inboundOfferCandidates` — `loanOfferCandidates` finds the single most-interested AI club per listing (`perceivedValueToClub` must clear the player's worth to the user's own club by `LOAN_MIN_SURPLUS`, a much looser bar than a permanent sale's `AI_MARKET_MIN_SURPLUS` since a loan is cheap and reversible) — surfaced on a new `/loans` page (sidebar under Team, next to Incoming Offers) with Accept/Reject only, no haggling, since the fee is flat by design. A reject is remembered per-window (`LeagueStore.loanRejections`) so the same buyer doesn't reappear immediately.
- **AI↔AI loans**: `runAILoanMarket`, structurally a slimmed-down mirror of `runAITransferMarket` (same seller-reservation/buyer-surplus/descending-surplus-order/live-guard pattern), restricted to players ≤ `LOAN_AI_MAX_AGE` (23) — the whole feature is developmental, so AI-initiated loans (both this and offering on the user's listings) only ever concern prospects, never established stars. **A player in his club's starting XI is never loaned out (2026-07-18, user decision: "only young players who don't get minutes on their regular team should be loaned out")** — `runAILoanMarket` computes each seller's `resolveXI` and skips its members, so "buried" is literal roster position, not just the valuation screen; this is also what keeps elite youngsters out of the loan pool (a 75+ prospect starts somewhere), closing the "80-OVR loanee parked in Division 2 all season, untouchable by the ceiling sweep" leak found in the same world-scale audit as the market guards below. The user's own manual listings are unrestricted (user manages manually, per convention). Runs once per window: inside `simOffseason` for the summer window (step 6.46, after the permanent AI↔AI market and the Division 2 ceiling sweep), and inside `simThrough`'s existing `winterMarketRunSeason`-guarded block for the winter window. The user's club never participates on either side of this automatic step — loaning the user's own players in/out is exclusively a manual action via the Loans page.
- **Returns**: `processLoanReturns` runs early in `simOffseason` (step 1.5, right after releasing expired contracts, before progression/free agency/season-start wage charging) — any loan whose `returnSeason` has arrived moves the player back to his parent club's roster, regardless of the parent's current roster size (mirrors how youth intake can briefly push a club over `ROSTER_CAP` — enforced at the next acquisition, not on return). No early recall in this pass — a loan runs its full agreed length; this is a disclosed scope cut, not an oversight.
- **Schema**: `LeagueStore` gained `activeLoans`/`loanListings`/`loanRejections` (all migrated to `[]`); `CompletedTransfer` gained optional `loanSeasons`/`loanReturn` fields (no migration needed — additive/optional, so old array entries just read `undefined` = a permanent transfer) so loan moves and returns show up in the News Feed and Finance transfer history like any other deal.
- **Deliberately out of scope for this pass** (disclosed, not hidden): no "Loan Market" browsing page for the user to loan a player *in* from an AI club — the user's own explicit ask only described the listing/incoming-offers flow (list a player, review AI offers), so that's what got built; the AI↔AI loan step covers the "full AI-AI loans" breadth the user asked for without needing a user-facing loan-in UI. Worth adding later if requested.

## Continental Cup (cross-country knockout tournament)

Per the user's ask for a tournament: a 16-team single-leg knockout played *alongside* the league season, contested by the top four clubs of each of the four tier-1 leagues (England/Spain/Italy/Germany) — 4×4 = 16, a clean power-of-2 bracket, no byes. Design brainstormed with the user in-conversation (no separate spec doc). New `src/core/cup/` module (`types.ts`, `cup.ts` pure bracket/qualify/seed logic, `simCup.ts` match resolution, `cupStats.ts` derived per-player stat lines).

- **Qualification is purely by league position, not OVR** (user's explicit call): the top `CUP_TEAMS_PER_LEAGUE` (4) of each tier-1 table qualify. Seeded 1–16 by finishing rank first (every champion outranks every runner-up, etc.), then league points within a rank tier, then into a standard single-elimination bracket (`seedOrder`, keeps top seeds apart). **Cup skips season 1** (no prior table to qualify from): the cup is seeded each offseason from the just-finished tier-1 tables (`buildCupState`, called in `simOffseason`), so season 1 has `league.cup === null` by construction and the first cup runs in season 2. `worldHasCup(competitions)` gates everything on the world fielding exactly a 16-team bracket (4 tier-1 leagues), so England-only legacy saves get no cup and no misleading UI.
- **Rounds fire on fixed league matchdays** (`CUP_ROUND_MATCHDAYS = [8,16,26,34]` for R16/QF/SF/Final), threaded through `simThrough`'s matchday loop the same way the winter transfer market is (a `dueCupRound` check per matchday). Each round uses its **own seeded rng** (`hashInts(lid, season, round, 30)`) so cup matches never perturb the league match stream — adding the cup leaves league results bit-identical for a given seed. A level tie plays extra time (`CUP_ET_CHANCES_PER_SIDE` shots/side via the existing `resolveShot` cascade, attributed to picked shooters/assisters + the defending keeper) then a penalty shootout (`playShootout`, best-of-5 then sudden death); every tie yields a winner. `resolveCupTie`/`playCupRound` in `simCup.ts`.
- **Stop-before-final**: if the user's club reaches the final (known once the semis are played), `simThrough` halts the batch *before* the final's matchday (`matchday > currentMatchday` so a *resumed* batch starting on the final's matchday actually plays it) — the unplayed matchdays fold back into `remaining`, phase stays `regular`. The Dashboard shows a warning banner so the user knows why the sim paused. Verified live in-browser end to end (extra time + shootout paths both exercised in real play; a completed season-2 cup crowned a champion).
- **Prize money** (title-ish, tuned with the user): credited to budgets in `simThrough` as each round is played (clamped by `clampBudget`). Participation £2M to all 16, then round-win bonuses (`CUP_PRIZE_WIN_BY_ROUND`) escalating to a £30M final-win — a champion nets ~£48M total, runner-up ~£24M, SF-loser ~£10M. Prize-only (never subtracts), so AI solvency is unaffected; a dynasty-scale audit of the exact magnitudes is a sensible follow-up but wasn't blocking.
- **Cup stats are tracked separately from league `SeasonStats`** (user's call), living only on each tie's stored `BoxScore` — they don't touch Stat Leaders, awards, or progression. A player's cup line is derived on demand (`cupStatsForPlayer`/`cupStatsBySeasonForPlayer`) and surfaced as a **Cup tab** on the Player Profile (alongside the existing League stats).
- **Standings** emphasize the top-4 qualification zone (`.cup-qualification` wash + `.cup-qualification-cut` cutoff border + a legend), declared before the user/champion row rules so those richer highlights win on overlap (champion keeps its gold, your club keeps its blue) while the other qualification slots stay shaded. Only shown for tier-1 tables in worlds that field a cup.
- **Schema**: `LeagueStore` gained `cup: CupState | null` (current season) and `cupHistory: CupState[]` (archived at offseason rollover), both migrated to `null`/`[]` for old saves. New **Continental Cup page** (`/cup`, sidebar under League) renders the live bracket (rounds as columns, user's club highlighted, champion banner, extra-time/pens notes) with a season dropdown over past cups. Manual gained a "The Continental Cup" section.
- Verified: `test/core/cup.test.ts` (10 pure bracket/qualify/seed/prize-accounting tests) + `test/core/cupIntegration.test.ts` (2 real-season tests: full season→offseason→cup lifecycle with completion + archive, and the stop-before-final halt/resume). Full suite otherwise green, typecheck + production build clean, browser-verified.

## Scouting fog-of-war on potential

Per the user's ask (a BBGM-style feature the "known gaps carried forward" list above used to flag as unaddressed: potential was previously shown to the user fully accurate). The scouting-spend slider — which until now only noised *transfer valuations* — also governs how precisely the user sees each player's **potential (POT)**. Design forks were confirmed with the user up front (AskUserQuestion): fog **POT only** (current OVR/attributes stay exact); present it as a **low–high range**; **all** players fogged with the user's own senior-roster players **clearing over ~2–3 seasons** of ownership; scouting spend **both** narrows the band and speeds clearing. AI clubs are unaffected — they already have their own `perceivedValueToClub` noise (AI GM phase 5); this is purely the user's *view*.

- **Core logic**: `potentialFog(potential, pid, currentSeason, observedSeason, scoutingSpend)` (`src/core/scouting/potentialFog.ts`) returns `{ low, high, known }`. Two independent axes both narrow the band from `SCOUT_POT_FOG_HALFWIDTH_MAX` (8) toward `_MIN` (2) and shorten the clear horizon from `SCOUT_POT_CLEAR_SEASONS_MAX` (3) to `_MIN` (2): scouting spend (interpolated on the existing `SCOUTING_SPEND_MIN`/`MAX` range) and tenure (`currentSeason − observedSeason`). At/after the clear horizon (or once the half-width rounds below 0.5) it returns `known: true` with `low === high === potential` (exact). The band's **center** is jittered off the true value by up to ±halfWidth × `SCOUT_POT_FOG_SHIFT_FRACTION` (0.5) — deterministic per `(pid, currentSeason)` via `mulberry32(hashInts(pid, currentSeason, salt))` so it doesn't flicker on re-render but *does* wobble year to year — while the band is guaranteed to still bracket the true value after rounding/clamping (so the midpoint is never trivially the answer, but the truth is never outside the shown range). All constants live in the "Scouting fog-of-war on potential" block of `src/core/constants.ts`.
- **Tenure tracking**: a new `StoredTeam.scoutingObserved: Record<pid, season>` (pid → the season the player was first seen on the user's *senior* roster), only ever populated for the user's own club (empty `{}` for every AI team). Maintained by a pure `reconcileScoutingObserved(prev, rosterPids, season)` reducer — keep existing entries for players still present, stamp new arrivals with the current season, drop departed pids (so a re-signing re-fogs). Called at league creation (`createLeagueState`, season 1, initial squad) and once per offseason (`simOffseason`, `nextSeason`, on the finalized new-season roster). **No acquisition site (`executeTransfer`/`signFreeAgent`/`promoteFromAcademy`) was touched**: a mid-season buy simply reads as tenure 0 (max fog) until the next offseason reconcile picks him up — correct behavior (a just-acquired player *is* unknown), and it keeps the wiring to two call sites. Prospects, free agents, academy players, and rival clubs' players are never on the senior roster, so `observedSeason` is `null` for them → always tenure 0 (foggiest for the current spend). Migrated to `{}` for old saves (existing squads re-fog once and clear over the next few seasons, an accepted one-time effect on load).
- **UI**: a single context-consuming `<PotDisplay player={p} />` (`src/ui/components/PotDisplay.tsx`) replaces every raw `{p.potential}` render — it pulls the user's `scoutingSpend`/`scoutingObserved`/`league.season` from `LeagueContext` and renders the exact number when known, else a dotted-underlined italic `low–high` estimate (`.scouting-estimate` in `styles.css`) with a hover title explaining the mechanic. Wired into: Roster (pitch chips + popover + bench table, via `PitchField.tsx`), `PlayerRatingsTooltip` (so every hover tooltip app-wide is consistent), Academy, Incoming Talent, Free Agents, Transfers (Recommended), Incoming Offers, Season Preview (top-players list), Power Rankings, and the Player Profile header. The Player Profile's **OVR/POT/attribute history table** also fogs its POT column, computed **per-row keyed off that row's own season** (not `league.season`) — this closes the "read the exact number one tab over" leak for a player the user hasn't scouted, while an owned player's history clears naturally with tenure. Team-level OVR/POT aggregates (Standings, Roster header, Season Preview's top-teams table) are left **exact** — they're rollups, not per-player scouting-decision surfaces (documented, minor: they're derived from the true per-player potentials).
- **Manual**: the "Players: Ratings, OVR & Potential" section gained a fog-of-war paragraph, the Finance/scouting section notes the second effect of the slider, and the FAQ gained a "why is potential shown as a range?" entry. The Finance page's scouting-slider help text was updated too.
- Verified: `test/core/scouting/potentialFog.test.ts` (11 tests: always brackets the truth, determinism, year-to-year wobble, spend narrows the band, monotone clearing to exact over tenure, spend speeds clearing, never-observed == brand-new, plus the `reconcileScoutingObserved` reducer's stamp/keep/drop cases); the touched `offseason`/`migrate`/`injuries`/`promotion`/`simThrough` suites all pass unchanged after adding the field to their fixtures; `tsc --noEmit` clean; production build clean.

**Scouting spend is locked per season (2026-07-18)**, closing a user-reported peek exploit: because the fog (and the M6 transfer-valuation noise) read `scoutingSpend` *live*, a user could crank the slider to max, read the narrowed POT ranges / sharper valuations across the app, then drag it back to £0 before season-end settlement and pay nothing for the info. Fixed by decoupling the *committed* value that drives fog/noise/charging from the value being *edited*: a new `StoredTeam.nextScoutingSpend` (migrated from `scoutingSpend` for old saves) holds the level the user sets for the *upcoming* season, editable **only during the offseason phase** (`setScoutingSpendAction` no-ops otherwise, and the Dashboard/Finance sliders are disabled during the regular season). `scoutingSpend` — which `potentialFog`/transfer noise/`settleSeasonEnd` all still read unchanged — is only updated (to `nextScoutingSpend`, clamped to budget) at the offseason rollover in `simOffseason` (replacing the old reset-to-`SCOUTING_SPEND_DEFAULT`). So dragging the slider can never sharpen the *current* view: you commit and pay first, then get the sharper view once that season starts. Charging stays at season-end (`settleSeasonEnd`) and AI teams are unaffected (they never touch `nextScoutingSpend`, so it stays at their creation-time default and `scoutingSpend` still resolves to the default each offseason exactly as before). Season 1 has no preceding offseason, so it runs at the default, locked — first adjustable in the offseason after season 1. Manual's Finance/scouting section + the "plan a year ahead" strategy tip updated to match. `test/core/offseason.test.ts` gained a lock-carry test; the old "resets scouting to default" test was retargeted to the new "locks in nextScoutingSpend" semantics.

## OVR history chart (Player Profile)

Per the user's ask ("add a graph like this [a Transfermarkt-style market-value area chart] but with OVR as the y axis... differentiate between academy and senior team"). A per-player career OVR-over-time area chart on the Player Profile page — line + gradient fill, with club crests marking transfers. Design forks confirmed with the user up front (AskUserQuestion): track academy/senior **going forward** (not reconstructed), include **club crests** at transfers, x-axis by **season**. **The line is colored by the club the player was at each season** (a follow-up user change from the original blue-academy/red-senior scheme) — it changes color when he transfers, and the academy/senior distinction now lives *only in the hover tooltip* ("Club (Academy)" vs just "Club"), not in the line color.

- **The one real data decision**: nothing previously recorded, historically, whether a player was in the academy or the senior squad in a given past season (academy only exists for the *user's* own club; there's no per-season roster snapshot). Rather than reconstruct, a per-season academy flag is now stored directly on each rating snapshot: `RatingsSnapshot.academy: boolean` (`src/core/players/types.ts`). It's populated from the 4th arg to `progressPlayer` — which already carried academy membership (`academyPids.has(p.pid)` from `simOffseason`, previously used only for the full-minutes assumption; the param was renamed `fullMinutes`→`inAcademy` for honesty). So every offseason's new snapshot records the truth going forward. Old saves migrate to `academy: false` (all-senior) in `src/db/migrate.ts` — past academy periods are irreconstructable, so a pre-existing youth product shows senior until new seasons accrue (same forward-only limitation as the scouting-tenure and per-season-team features).
- **The generation baseline snapshot** (`generatePlayer`'s single `hist` entry at `season - 1`) defaults `academy: false`, but `simOffseason` step 5 flips it to `true` for the user's youth intake at the same spot it stamps their academy contract — otherwise the hover tooltip would read "senior" for a youth product's very first snapshot before his first real academy season. (The academy flag no longer affects line color, only the tooltip label; it stays useful for that.)
- **Chart component**: `src/ui/components/OvrHistoryChart.tsx`, pure SVG (viewBox space, rendered at width:100%/height:auto so a fixed aspect keeps the crest overlay aligned), everything derived from `player.hist` (per-season OVR + academy flag) and `league.transfers` — no stored chart state, OVR always exact (fog is POT-only). **Club coloring**: `teamLineColor()` picks each club's *primary* color, lightening it toward white if it's too dark to read on the dark surface (a near-black primary falls back to the secondary), so the line keeps club identity while staying visible; `readableText()` picks black/white badge text for contrast. The line is split into colored runs by the club-per-season, and a transfer *between* two seasons splits that segment **at its midpoint** (departing club colors the first half, arriving club the second) so every season's club owns the half-segments next to its point and the line color always matches the point's dot. OVR y-axis with nice-number gridlines, season-year x-axis (thinned to ≤8 labels), and transfer crests HTML-overlaid (reusing `ClubCrest`, positioned by % so the art scales at a fixed size). **Interactive hover** (the component's only local state): the cursor snaps a vertical crosshair + enlarged club-colored dot to the nearest season and shows a floating pill-card tooltip — crest + the club the player was on *that season* (with " (Academy)" appended for an academy year) + season year + an OVR badge in the club's color. The tooltip anchors left/center/right by the point's horizontal position so it never clips the chart edges. The club-per-season comes from a `teamTidForSeason` prop wired from `PlayerProfile`'s existing `teamForSeason` transfer-walk (falling back to "Free agent"). Multiple transfers in one season (a summer + winter deal, or an AI journeyman's churn) share an anchor, so same-season crests are fanned horizontally — centered on the season and clamped to the plot width so a busy season never overflows the chart edges — rather than stacking on one spot. CSS in the "OVR history chart" block of `styles.css`. Wired into `PlayerProfile.tsx` as an "OVR History" card above the existing OVR/POT/attribute history table.
- Browser-verified end to end on a real 3-season dynasty: an academy youth renders a single club-colored line and a "Club (Academy)" hover card; a well-travelled player's line tiles cleanly through each club's color with matching gradient fills, crests at transfer seasons, and a hover card naming the correct *historical* club per season. `tsc --noEmit` clean, production build clean; `progression`/`migrate`/`offseason` suites pass with the new field.

## Extremism (generational talents + historic team seasons)

Per the user's ask (2026-07-19) that the sim produce occasional *extreme* outcomes — the anti-inflation tuning documented above had (correctly) clipped every distribution's tails, but that also meant no legends and no "remember that season?" campaigns. Two independent mechanics, both derived deterministically from hashes (zero shared-rng consumption, zero schema change, zero migration — the `developmentBias` pattern):

- **Generational talents** (`isGenerational(pid)`, `src/core/players/progression.ts`): a tiny fraction of players (`GENERATIONAL_CHANCE`, 1/2500 per pid ≈ one new arrival somewhere in the world every ~4-5 seasons) develop under a much softer growth-damping curve (`GENERATIONAL_DAMPING_END`/`_FLOOR`: 95/0.6 vs the normal 80/0.02) with their development-bias z floored at `GENERATIONAL_BIAS_MIN_Z` (1.5) — never a total bust, though per-season form rolls still vary. Tuned empirically (200-trial career sims): announced kids peak at a median ~80 OVR, ~30% reach 85+, ~13% reach 90+ (max observed 96), while an identical unflagged kid never passed 79 — so a genuine 90+ legend emerges roughly once every ~30 seasons. `estimatePotential` flows through the same `stepRatings`, so scouts genuinely see the higher ceiling. Arrival via youth intake is announced on the News Feed (`generationalTalent` event type, matchday 0, detail = age; detected in `simOffseason` step 5 by checking the flag, not rolling anything). Players flagged at initial world-gen exist too but aren't announced (no feed exists at creation).
- **Historic team seasons** (`src/core/teamSeasonForm.ts`): each club-season has a `TEAM_SEASON_FORM_PROB` (1.5% each direction) chance of a hidden season-long form swing of ±`TEAM_SEASON_FORM_DELTA` (0.12, ≈ ±1.5 z of true squad strength) applied to all five normalized composites when `simThrough` builds each matchday's `TeamMatchData` — wrapped around `recompute` too (so subs/red cards keep the offset), and the cup shares the same matchData map so cup ties feel it as well. Derived from `hash(lid, season, tid)`: survives any sim batching, re-rolls each season. Ratings/valuations/wages untouched — purely on-pitch, so standings/hype/prize money respond naturally with no market side effects. The user's club is eligible like any other (explicit user call: symmetric, both directions). **Tuned against a form-disabled control on the same seed** (important: the baseline game *already* organically produces ~3 champions with 90+ points and ~24 bottom clubs with ≤15 points per 240 division-seasons — any future audit of this feature must compare against a control rather than crediting all extremes to it): the first-guess delta of 0.06 was a table-level no-op (5 such champions vs the organic 3); 0.12 lands 8 with a best of 100 points, while champion churn (14-18 distinct D1 winners/30 seasons) and the OVR equilibrium hold.

**Headless-audit lesson (2026-07-19), learned chasing the extreme bottom-club seasons the audits above surfaced**: the catastrophic ≤10-point seasons (worst 0-2 points) are **entirely the user's own club rotting unmanaged** — a corpse-club autopsy across 30 seasons found every single ≤10-point club-season was tid = userTid (roster shrunk to 15-18, top-11 OVR in the high 20s, 10-14 teenagers fielded), because by long-standing design nothing manages the user's roster in a headless sim (no AI free agency/renewals/buys for the user; `ensureUserRosterSafety` only refills to 18 bodies with academy kids). **AI clubs never fall below ~11-15 points** — a genuinely realistic floor (Derby's real record is 11). Two consequences: (1) an attempted fix (flooring Division 2's generation band + an academyBase healing drift) was built, audited, found to move nothing — the diagnosis it targeted was wrong — and **fully reverted**; the "1-point seasons are structural D2 weakness" theory in earlier drafts of this section is falsified. (2) **Every future dynasty audit must exclude the user's club from points/roster-extreme metrics** (or report it separately) — its unmanaged decay contaminates minima, bottom-club stats, and any per-club tail analysis, while aggregate OVR means (500+ players) are only negligibly affected.

Manual updated (Player Development gained a "Generational talents" paragraph; The Season & Simming gained a "Historic seasons" paragraph — behavior described, hidden values unspoiled — and its stale "there are no cups yet" line was fixed in passing). Verified via a 30-season world audit + the control run above: OVR equilibrium intact (D1 mean ~59-62, 80+ 1-3%, ordinary 90+ 0%), generational arrival rate on target (4 announced/30 seasons, peaks 88/84/78/75 with two still developing), and a 100-point title campaign now genuinely possible.
